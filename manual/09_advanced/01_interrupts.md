## Режимы прерываний Z80

Z80 поддерживает 3 режима прерываний:

| Режим | Инструкция | Поведение |
|-------|-----------|-----------|
| IM 0 | `IM 0` | Устройство кладёт опкод на шину, CPU исполняет |
| IM 1 | `IM 1` | При прерывании → `RST #38` |
| IM 2 | `IM 2` | Векторный режим: I × 256 + vector |

В Sprinter чаще всего используется **IM 2** — он позволяет разделить прерывания от разных источников (VSync, CTC, SIO).

---

## Источники прерываний

В Sprinter существует несколько аппаратных источников прерываний:

| Источник | Частота | Сигнал в FPGA | Описание |
|----------|---------|--------------|----------|
| **Кадровое (VSync)** | ~50 Гц (PAL) | `SVIDEO.INTT` | Конец кадра — основной таймер |
| **Клавиатурное (PS/2)** | ~2.1 кГц | `KEYS.int` | Счётчик бит PS/2 достиг 0 |
| **CTC** (4 канала) | настраивается | через Z84C15 | Пользовательские таймеры |
| **SIO Rx** | по событию | через Z84C15 | Получен байт от PS/2 |
| **CBL (COVOX)** | настраивается | `CBL_INT` | Буфер аудио опустошён наполовину |
| **NMI** | по событию | `NMI_X` | Alt+F12 (если включено в прошивке) |
| **ISA** | по событию | через ISA-слот | IRQ от ISA-устройства |

## Формирование сигнала INT — аппаратная схема

Сигнал `/INT` процессора Z80 формируется в FPGA (`SP2_ACEX.TDF`) из **трёх** независимых источников:

```
INT_X = !DFF(GND, (SVIDEO.INTT & KEYS.int), ...) or !CBL_INT;
```

Разберём подробно:

### 1. Кадровое прерывание (SVIDEO.INTT)

Генерируется в видеоконтроллере (`VIDEO2.TDF`) при достижении определённой позиции растра:

```
INTTX = DFF((BORD & MODE03 & MODE02 & MODE00), LWR_COL,,);
INTT  = DFF(!(INTTX & (CTV[2..0] == 7)), CT5,,);
```

Срабатывает **один раз за кадр** (~50 Гц PAL). Привязано к позиции бордюра — аналогично ZX Spectrum.

### 2. Клавиатурное прерывание (KEYS.int)

Генерируется в модуле клавиатуры (`KBD.TDF`):

```
INT = DFF((KB_CT[] == 0), CLK42,, INT_ENA);
```

`KB_CT` — счётчик бит PS/2 (7→0), тактируемый CLK_K (~15 кГц). Прерывание возникает каждые **~460 мкс** (7 тактов × 67 мкс). Управляется через `INT_ENA`:

```
; SP2_ACEX.TDF, строка 325:
KEYS.int_ena = LCELL(ALL_MODE0 & ALL_MODE3);
```

Клавиатурное прерывание **включено только** при `ALL_MODE0 = 1` **и** `ALL_MODE3 = 1` (регистр `ALL_MODE`, порт `#C3`).

### 3. COVOX/CBL прерывание (CBL_INT)

Генерируется при опустошении половины FIFO-буфера COVOX:

```
CBL_INT = DFF(GND, !CBL_CNT6,, (CBL_INT_ENA & (/IO or /M1)));
```

Включается битом 4 порта `#89` (`CBL_INT_ENA`).

### Итоговая формула

Все три источника объединяются в один сигнал `/INT`:

```
INT_X = !DFF(GND, (SVIDEO.INTT & KEYS.int), acknowledge) or !CBL_INT;
```

Обратите внимание: кадровое и клавиатурное прерывания соединены через **AND** — это ключевая особенность, порождающая проблему нестабильных прерываний.

## CBL (COVOX) прерывание

Включается битом 4 порта `#89`:

```asm
        ld      a, #9C              ; COVOX ON + INT enable + 31.25 кГц
        out     (#89), a
```

Прерывание приходит при опустошении половины 256-байтного FIFO. В обработчике программа дозаписывает следующую порцию сэмплов. Подробнее — в разделе [6.3 COVOX DMA](../06_audio/03_covox_dma.md).

## NMI

Non-Maskable Interrupt в Sprinter:

```
; SP2_ACEX.TDF:
IF (NMI_ON == "ON") GENERATE
    NMI_X = (!KEYS.kb_f12 & KEYS.kb_alt);   ; Alt+F12
END GENERATE;
```

По умолчанию NMI **отключён** в прошивке (`NMI_ON = "OFF"`). Если включён — срабатывает по Alt+F12, переход на `RST #66`.

## IM 1 — простой режим

При IM 1 все прерывания идут на `RST #38` (адрес `0x0038`).

```asm
        im      1
        ei

; Обработчик по адресу 0x0038
        org     #0038
        push    af
        ; ... обработка ...
        pop     af
        ei
        reti
```

Проблема IM 1: невозможно различить источники — одна точка входа для всех прерываний.

## IM 2 — векторный режим

В IM 2 Z80 вычисляет адрес обработчика по формуле:
```
Vector_addr = (I << 8) | Vector_byte
```

где `I` — регистр I (старший байт), `Vector_byte` — значение, которое устройство кладёт на шину данных при подтверждении прерывания.

По этому адресу лежит **16-битный указатель** на обработчик. Z80 читает его и переходит туда.

### Проблема в Sprinter

В IM 2 при подтверждении прерывания устройство кладёт на шину данных **вектор-байт**. Z80 читает 16-битный указатель из адреса `I × 256 + вектор`. Проблема: разные устройства (CTC канал 0, канал 1, SIO, FPGA) кладут **разные** значения вектора. Если выделять отдельный обработчик на каждый вектор — нужна сложная диспетчеризация.

**Трюк с таблицей 257 одинаковых байт** решает проблему: таблица заполняется одним и тем же значением (например, `#FD`). Тогда **любой** вектор-байт, считанный с шины, даст один и тот же адрес обработчика.

Почему 257 байт, а не 256? Потому что вектор-байт может быть нечётным — тогда Z80 читает слово, начиная с нечётного адреса. Если на границе таблицы (позиция 255) будет `#FD`, а байт 256 — мусор, получится неверный адрес. Лишний 257-й байт гарантирует, что чтение с позиции 255 даст `#FDFD`, а не `#FD??`.

Пример: `I = #FE`, таблица на `#FE00`–`#FF00` заполнена `#FD`. Любой вектор `V` → адрес `#FE00 + V` → читается слово `#FDFD` → переход на обработчик по адресу, записанному в `#FDFD`.

## Стандартная настройка IM 2 в Sprinter

### Таблица векторов на #FE00

```asm
        ; 1. Заполнить 257 байт значением #FD
        ld      hl, #FE00
        ld      de, #FE01
        ld      bc, 256             ; 256 + 1 = 257 байт
        ld      (hl), #FD
        ldir

        ; 2. По адресу #FDFD записать указатель на наш обработчик
        ;    (Z80 прочитает слово #FDFD → int_handler)
        ld      hl, int_handler
        ld      (#FDFD), hl

        ; 3. Установить I = #FE → таблица начинается с #FE00
        ld      a, #FE
        ld      i, a

        im      2
        ei
```

Как это работает:

```
Любое устройство кладёт вектор V (0..255) на шину данных
    ↓
Z80 вычисляет: адрес = I × 256 + V = #FE00 + V
    ↓
По адресу #FE00+V лежит байт #FD
По адресу #FE00+V+1 тоже #FD
    ↓
Z80 читает 16-битное слово: #FDFD
    ↓
Z80 переходит на адрес, записанный в (#FDFD): JP int_handler
```

### Обработчик

```asm
int_handler:
        push    af
        push    bc
        push    de
        push    hl
        push    ix
        push    iy
        ex      af, af'
        push    af
        exx
        push    bc
        push    de
        push    hl

        ; ... основная обработка ...
        call    VSyncHandler

        pop     hl
        pop     de
        pop     bc
        exx
        pop     af
        ex      af, af'
        pop     iy
        pop     ix
        pop     hl
        pop     de
        pop     bc
        pop     af
        ei
        reti
```

## Приоритеты прерываний

Z80 не имеет аппаратного приоритета — прерывания обрабатываются в порядке поступления, одно за раз. Если во время обработки приходит новое — оно ждёт `EI` + `RETI`.

**Вложенные прерывания** (reentrant) возможны только если в обработчике явно сделать `EI`, но это требует тщательного управления стеком.

## Пример: полный каркас программы с IM2

```asm
        org     #8100

main:
        di

        ; Заполнить таблицу IM2
        ld      hl, #FE00
        ld      de, #FE01
        ld      bc, 256
        ld      (hl), #FD
        ldir

        ; Указатель обработчика на #FDFD
        ld      a, #C3              ; JP opcode
        ld      (#FDFD), a
        ld      hl, int_handler
        ld      (#FDFE), hl

        ld      a, #FE
        ld      i, a
        im      2
        ei

        ; Главный цикл
.loop:
        halt                        ; ждать VSync
        call    GameLogic
        jr      .loop

int_handler:
        push    af
        push    hl

        ld      hl, (tick_count)
        inc     hl
        ld      (tick_count), hl

        pop     hl
        pop     af
        ei
        reti

tick_count:     dw 0

GameLogic:
        ; ... игровая логика ...
        ret
```

## Ключевые моменты

> - **Источники INT:** VSync (~50 Гц), клавиатура PS/2 (~2.1 кГц), CTC (настраиваемый), CBL (COVOX), NMI (Alt+F12).
> - **Формула INT:** `INT_X = !DFF(GND, (SVIDEO.INTT & KEYS.int), ...) or !CBL_INT` — клавиатура влияет на тайминг кадрового прерывания (джиттер до 460 мкс).
> - **IM 2 трюк с 257 байтами:** таблица из одинаковых `#FD`; любой вектор → `#FDFD` → единый обработчик.
> - CTC, CBL, NMI — независимые источники с собственными условиями срабатывания.
> - Всегда сохраняйте регистры в обработчике, `EI + RETI` в конце.
> - **Практика** (стабилизация VSync, tearing, SDK обработчик) → [9.5 Практика работы с кадровым прерыванием](05_vsync_practice.md).
