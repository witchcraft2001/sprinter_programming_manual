## Архитектура кэша в Sprinter

Sprinter не имеет традиционного CPU-кэша (L1/L2 в смысле сокрытия латентностей RAM). Вместо этого, термин "кэш" в документации Sprinter обозначает **небольшой блок быстрой SRAM**, отображённый в WIN0 (`0x0000–0x3FFF`) вместо DRAM при включении специального режима.

Механизм управляется через **порт `#88`** и сигнал `CASH_ON` в FPGA:

```
Источник SP2_ACEX.TDF:
CASH_ON = DFFE(A7, (/IO or /RD), /RESET,, DFF((A[7..0] == B"X1111011"), CLK42,,));
```

Это означает: кэш включается при чтении через порт с маской `#7B` (или `#FB`).

> **Важно для программиста:** Кэш-механизм прозрачен и не требует ручного управления в большинстве программ DSS. Актуален только при прямом доступе к низкоуровневым портам или при разработке загрузчиков.

## Wait States — основная концепция тайминга

В Sprinter все обращения к памяти и портам вставляют **дополнительные такты ожидания** (`WAIT`). Это необходимо потому, что DRAM медленнее, чем Z80 на 7 МГц, и особенно на 21 МГц turbo.

Количество wait states определяется в `DCP.TDF` через таблицу `W_TAB`:

### Нормальный режим (7 МГц)

| Тип памяти/порта | Wait states | Эффективная скорость |
|----------------|-------------|---------------------|
| RAM (страницы `#00`–`#3F`) | 2 | ~4.7 МГц |
| ROM/Flash | 2 | ~4.7 МГц |
| VRAM (страницы `#50`–`#5F`) | 7 | медленно |
| MAX7000 (FDD, HDD регистры) | 10 | очень медленно |
| IDE данные (`#50`/`#150`) | 4 | — |

### Turbo режим (21 МГц)

| Тип памяти/порта | Wait states |
|----------------|-------------|
| RAM | 2 |
| ROM/Flash | 4 |
| VRAM | 7 |
| MAX7000 / HDD регистры | 10 |
| IDE данные | 4 |

> Wait states в turbo-режиме не уменьшаются пропорционально: ROM требует в 2 раза больше тактов, VRAM и MAX7000 — те же абсолютные числа. Реальный прирост производительности от turbo зависит от типа используемой памяти.

## Wait states при видеоконфликтах (ZX-режим)

В **ZX Spectrum-совместимом режиме** (`SC0=1`) видеоконтроллер и Z80 конкурируют за доступ к VRAM:

- Когда Z80 обращается к странице `#05` (ZX-экран, WIN1 = `0x4000`–`0x7FFF`), аппаратура вставляет до **10 дополнительных тактов** в момент, когда видеоконтроллер читает ту же память.
- Это поведение идентично оригинальному ZX Spectrum 48K (contended memory).

В нативном Sprinter-режиме (`SC0=0`) видеоконфликтов нет: VRAM и DRAM разделены физически (разные чипы).

## DRAM-контроллер

DRAM (512 КБ на базовой плате) управляется через **RAS/CAS** сигналы FPGA. DCP формирует:

- `RAS` — Row Address Strobe
- `CAS` — Column Address Strobe
- `MC_BEGIN` / `MC_END` — начало и конец цикла обращения к памяти

Временная диаграмма цикла чтения:

```
CLK42   ___/‾‾‾\___/‾‾‾\___/‾‾‾\___/‾‾‾\___/‾‾‾\___
/MR     ‾‾‾‾‾\___________________________/‾‾‾‾‾‾‾‾‾‾
MC_BEGIN      \______/
MC_END                          \_____/
MC_TYPE ‾‾‾‾‾‾‾‾\_______________________/‾‾‾‾‾‾‾‾‾‾
```

При `DOUBLE_CAS=1` (вход ACEX) — режим двойного CAS для Alliance VRAM (улучшенная совместимость, версия прошивки 3.04).

## Оптимизация производительности

### Избегать обращений к VRAM в критическом коде

```asm
; ПЛОХО: частое чтение/запись VRAM замедляет код
loop:   ld      a, (VRAM_addr)
        inc     a
        ld      (VRAM_addr), a
        djnz    loop

; ЛУЧШЕ: работать в RAM, копировать во VRAM блоком
        ; 1. Сгенерировать данные в RAM
        ; 2. Скопировать в VRAM через акселератор (blitter)
        ; 3. Flip double buffer
```

### Использование турбо-режима

В turbo (21 МГц) код выполняется в 3 раза быстрее — но только для инструкций из RAM. Обращения к ROM добавляют 4 wait state вместо 2:

```asm
; Цикл из RAM — получает полное ускорение 3×
; Цикл с обращением к ROM — медленнее из-за wait states

; Лучшая практика в turbo: держать горячий код в RAM,
; не вызывать BIOS/DSS RST часто в узких циклах
```

### Блочные операции с DRAM

Инструкция `LDIR` для копирования RAM→RAM эффективна. Каждый байт: M1 цикл (1 wait) + RD + WR циклы (по 2 wait каждый):

```asm
; Копировать 16 КБ за один LDIR: ~100–120 мс при 7 МГц
        ld      hl, src_addr
        ld      de, dst_addr
        ld      bc, #4000
        ldir
```

## Аппаратный кэш-буфер (CASHE)

В SP2_ACEX.TDF присутствует логика `CS_CASH` и `CS_CASHT` — дополнительный блок кэша для страниц `#F0`–`#FF` в специальном режиме:

```
PRE_CASH  = LCELL(A14 or A15 or !CASH_ON);
CS_CASHT  = DFF((!/RF or !PRE_ROM or PRE_CASH), !/MR,, LCELL(CS_CASHT or !/MR));
CS_CASH   = CS_CASHT & CS_ISA & ISA_A20;
```

Это аппаратная функция, **недоступная напрямую** из пользовательского кода.

## Влияние wait states на тайминги

Реальная эффективная тактовая частота Z80 (с учётом wait states):

| Операция | Такты при 7 МГц | Эффект. скорость |
|---------|-----------------|------------------|
| LD r, r (регистр) | 4 | 1.75 МГц |
| LD r, (HL) — RAM | 4+2 = 6 wait | 1.17 МГц |
| OUT (n), A — RAM | 10 такты | — |
| OUT (n), A — MAX7000 | 10+10 wait | очень медленно |

Пример: цикл записи в FDD через MAX7000 (10 wait states):

```asm
; Каждый OUT занимает ~10 мкс при 7 МГц + 10 wait states
FDD_WRITE_LOOP:
        out     (#FF), a        ; WG93 команда — ~24 такта = ~3.4 мкс
        djnz    FDD_WRITE_LOOP
```

---

## Ключевые моменты

> - Основное ограничение производительности — wait states при обращении к медленной памяти.
> - RAM: 2 wait states; ROM: 2 (normal) / 4 (turbo); VRAM: 7; MAX7000/IDE: 10.
> - В ZX-режиме VRAM может добавлять до 10 wait states из-за видеоконфликтов.
> - Turbo ускоряет только CPU-инструкции, не сокращает wait states пропорционально.
> - Для оптимальной скорости: держать горячий код в RAM, использовать акселератор для работы с VRAM.
