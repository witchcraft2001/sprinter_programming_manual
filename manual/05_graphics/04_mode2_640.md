# 5.4 Режим 640×512, 4bpp

> **Навигация:** [← 5.3 Режим 320×256](03_mode1_320.md) | [Оглавление](../README.md) | [5.5 Палитра →](05_palette.md)

---

## Характеристики

| Параметр | Значение |
|---------|---------|
| Разрешение | 640 × 512 пикселей |
| Биты на пиксель | 4 (16 цветов) |
| Формат | 2 пикселя на 1 байт (старший ниббл = левый пиксель) |
| Размер экрана | 640 × 512 / 2 = 163 840 байт |
| Страниц VRAM 16 КБ | 10 страниц (#50–#59 — весь экран) |
| Double buffering | Невозможен в 512 КБ VRAM |

Этот режим предназначен для **текстовых и CAD-приложений** с высокой чёткостью. Размер экрана — весь объём VRAM, поэтому второго буфера нет.

---

## Включение режима

### Через DSS API

```asm
        ld      a, 2                ; 2 = 640×512 4bpp
        ld      b, 0
        ld      c, #50              ; SetVMod
        rst     #10
```

### Через прямые порты

```asm
        ld      a, 2
        out     (#C3), a            ; ALL_MODE = 640×512
```

---

## Формат пикселя

```
Байт: [Pixel_Left][Pixel_Right]
       7 6 5 4    3 2 1 0

   Старший ниббл — левый пиксель (индекс 0..15)
   Младший ниббл — правый пиксель
```

### Вычисление адреса

```
Адрес = BASE + (Y * 640 + X) / 2
Ниббл: (X mod 2 == 0) → старший (левый), иначе младший (правый)
```

---

## Установка пикселя

```asm
; Установить пиксель (X, Y) цвета A (0..15)
; Вход: DE = X, HL = Y, A = цвет (0..15)
SetPixel640:
        push    af

        ; Вычислить линейный адрес: Y * 640 + X
        ; Y * 640 = Y * 512 + Y * 128 = Y << 9 + Y << 7

        ld      h, 0
        add     hl, hl              ; Y * 2
        ; ... (здесь необходима 24-битная арифметика т.к. буфер > 64 КБ)

        ; Для простоты: расчёт через 16-битное переполнение с использованием
        ; прибавления строк по 320 байт (640/2)

        ; ... упрощённый вариант для небольших областей ...

        pop     af
        ret
```

**Замечание:** из-за большого размера буфера (160 КБ > 64 КБ Z80) эффективная работа в этом режиме **требует переключения страниц VRAM**. Строго линейный доступ через одно окно невозможен для всего экрана.

---

## Работа с буфером > 64 КБ

Для доступа к любому пикселю нужно:
1. Вычислить **линейный 18-битный адрес**.
2. Определить страницу VRAM: `page = #50 + (linear / 16384)`.
3. Определить смещение в окне: `offset = linear mod 16384`.
4. Подключить страницу в окно (обычно WIN3 = `0xC000`).
5. Обратиться по адресу `0xC000 + offset`.

```asm
; Установить пиксель (X, Y) в режиме 640×512
; Вход: DE = X (0..639), BC = Y (0..511), A = цвет (0..15)

SetPixel640:
        push    af
        ; Вычислить Y*320 в 18-битной арифметике
        ; 320 = #140

        ; HL = Y * 320
        ld      h, b
        ld      l, c
        push    hl                  ; сохранить Y
        ld      a, h
        ld      h, 0                ; расширенный H
        add     hl, hl              ; Y * 2
        add     hl, hl              ; * 4
        add     hl, hl              ; * 8
        add     hl, hl              ; * 16
        add     hl, hl              ; * 32
        add     hl, hl              ; * 64
        ; HL = Y * 64 (потенциальное переполнение)
        pop     bc                  ; BC = исходный Y
        ; ... полная 24-битная реализация опущена для краткости
        ; Используйте 24-бит через пару регистров + переполнение

        pop     af
        ret
```

> **Практика:** для работы в 640×512 обычно используют **построчный буфер** в RAM, который затем выгружается построчно в VRAM через acceлератор. Прямой поэпиксельный доступ слишком медленный.

## Рисование текста в 640×512

Этот режим идеально подходит для текста с разрешением **80 × 32 символа** (при шрифте 8×16):

```asm
; Напечатать 8-битный символ в позиции (col, row)
; Вход: B = col (0..79), C = row (0..31), A = символ, E = цвет
PrintChar640:
        ; Каждый символ занимает 8 пикселей по ширине (= 4 байта в 4bpp)
        ; и 16 пикселей по высоте
        ; Адрес начала: row*16*320 + col*4

        ; ... вычисление адреса ...

        ; Копирование 16 строк шрифта по 4 байта
        ld      b, 16               ; 16 строк символа
.row_loop:
        ; Прочитать байт шрифта (8 бит = 8 пикселей)
        ; Преобразовать в 4 байта 4bpp (старший ниббл = пиксель 0 цвета E, если бит=1)
        ; ...
        djnz    .row_loop
        ret
```

---

## Ограничения режима

1. **Нет double buffering** — невозможно избежать tearing без аккуратной синхронизации.
2. **Прямой доступ медленный** — каждая операция требует переключения VRAM-страниц.
3. **Только 16 цветов** из 256-цветной палитры.
4. **Acceлератор работает** в этом режиме, но с учётом 4bpp-упаковки.

---

## Когда использовать 640×512

- CAD-приложения, диаграммы.
- Векторная графика.
- Работа с высокодетализированной статичной графикой.
- Вывод отсканированных/растровых изображений.

Для **текстовых приложений** (редакторы, файловые менеджеры) используйте [текстовый режим 80×32](09_text_mode.md) — он значительно проще и быстрее.

Для **игр** предпочтительнее 320×256: double buffering и быстрый линейный доступ.

---

## Пример: текстовая программа

Для текстовых приложений (редакторы, файловые менеджеры) удобнее использовать встроенный **текстовый режим 80×32** на базе графического 640×256, а не поэпиксельно рисовать текст в 640×512. Именно так работает Sprinter TextEditor ([https://github.com/witchcraft2001/sprinter-texteditor](https://github.com/witchcraft2001/sprinter-texteditor)) — он использует DSS API `PChars #5C`, `PutChar #5B`, `Locate #52`. Подробности — в разделе [5.9 Текстовый режим 80×32](09_text_mode.md).

Режим 640×512 4bpp имеет смысл для:
- Статичной высокоразрешённой графики
- CAD/вычерчивания диаграмм
- Вывода растровых изображений с линейной разбивкой

---

## Ключевые моменты

> - 640×512 4bpp, 2 пикселя на байт, старший ниббл = левый пиксель.
> - Буфер 160 КБ = 10 страниц VRAM (`#50`–`#59`).
> - Нет double buffering — весь VRAM занят одним экраном.
> - Для доступа к пикселю: переключение страницы VRAM + расчёт смещения.
> - Оптимально для текста 80×32 и CAD-приложений.
> - Игры используйте в режиме 320×256.
