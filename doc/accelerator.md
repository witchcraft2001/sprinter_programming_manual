# Аппаратный ускоритель Sprinter

## Назначение

`ACCELER.TDF` реализует аппаратный DMA-ускоритель, который перехватывает специальные инструкции Z80 и выполняет блочные операции с памятью без участия CPU. Ускоритель может:

- Быстро заполнять области памяти константой
- Копировать блоки данных (горизонтально и вертикально)
- Работать в 16-битном режиме (двойной байт)
- Читать данные напрямую из IDE (режим DMA)
- Управлять Covox DMA (вместе с CBL-блоком)
- Выполнять логические операции (AND/OR/XOR) над данными

Ускоритель включается битом `ALL_MODE0=1` (порт `0xC3`). При `ALL_MODE0=0` блок полностью прозрачен.

## Интерфейс модуля

**Входы от Z80/DCP:**

| Сигнал | Назначение |
|--------|-----------|
| `AI[15:0]` | Адрес от Z80 (захватывается в AA при старте) |
| `DI[7:0]` | Данные шины Z80 |
| `CLK_Z80` | Тактовая частота Z80 |
| `/MR, /IO, /RD, /WR, /M1, /RF` | Управляющие сигналы Z80 |
| `MC_END/BEGIN/TYPE/WRITE` | Сигналы цикла от DCP |
| `RAS, CAS` | Стробы DRAM от DCP |
| `HDDR[7:0]` | Старший байт IDE-данных |
| `HDD_FLIP` | Признак 2-го байта IDE |
| `ACC_ENA` | Разрешение ускорителя (`ALL_MODE0`) |
| `DCP[7:0]` | Текущий порт (`DCPP[]`) |

**Выходы:**

| Сигнал | Назначение |
|--------|-----------|
| `AO[15:0]` | Адрес, который ускоритель подставляет вместо Z80 |
| `DO[7:0]` | Данные от ускорителя на шину |
| `MDO[15:0]` | 16-битные данные в DRAM |
| `CONTINUE` | Разрешение завершения цикла DCP |
| `DOUBLE_CAS` | Запрос удвоенного CAS-цикла |
| `ACC_ON` | Состояние ускорителя (= `ACC_DIR0`) |
| `ACC_DIR[7:0]` | Вектор режима операции |
| `G_LINE[7:0]` | Текущая позиция в кадре (для видеоконтроллера) |
| `GLISSER` | Признак байта 0xFF (белая заливка) |

## Декодирование инструкций Z80

Ускоритель отслеживает байты инструкций во время цикла выборки команды (`/M1` активен). Используются несколько триггеров:

### Префиксы (PRF_CMD)

```vhdl
PRF_CMD.d = (DI == 1100X01Xb) &
            ((DI == CB) or (DI == DD) or (DI == ED) or (DI == FD))
```

Если текущий байт — один из префиксов CB/DD/ED/FD, `PRF_CMD` устанавливается. Это предотвращает ложное срабатывание на опкоды с такими же кодами.

### Распознавание режима ускорителя (ACC_MODE)

Ускоритель активируется при инструкциях вида `LD r,r` (загрузка регистра в себя, что не имеет смысла в обычном коде):

```vhdl
ACC_MODE[].ena = DFF(
    !/M1 & !PRF_CMD &
    ((DI == XXX00X00b) or (DI == XXX01X01b) or
     (DI == XXX10X10b) or (DI == XXX11X11b)) &
    ((DI == 010XX0XXb) or (DI == 011XX1XXb)),
    CLK_Z80)
ACC_MODE[3:0].d = (1, DI[2:0])
```

Биты DI[2:0] кодируют режим, DI[5:3] должны совпадать с DI[2:0] (структура `LD r,r`).

### Таблица режимов ACC_MODE

| ACC_MODE[2:0] | Инструкция | ACC_DIR[7:0] | Назначение |
|--------------|-----------|--------------|-----------|
| 0 | `LD B,B` (0x40) | `00000000` | Выключен (нет операции) |
| 1 | `LD C,C` (0x49) | `00100101` | Заливка константой |
| 2 | `LD D,D` (0x52) | `00001001` | Загрузка счётчика |
| 3 | `LD E,E` (0x5B) | `00010101` | Вертикальная заливка |
| 4 | `LD H,H` (0x64) | `01000001` | Двойной байт (16-бит) |
| 5 | `LD L,L` (0x6D) | `00100111` | Копирование строки |
| 6 | `HALT` (0x76) | `00000000` | Принудительная остановка |
| 7 | `LD A,A` (0x7F) | `00010111` | Вертикальное копирование |

`ACC_MODE3` (старший бит) — признак «блокировка ускорителя при NMI»: при `1` устанавливается `ACC_BLK.prn`, что сбрасывает блокировку при `RETI`.

### Таблица битов ACC_DIR

| Бит | Название | Действие |
|-----|----------|----------|
| 0 | ACC_ON | Ускоритель активен (выводится наружу) |
| 1 | ACC_DIR1 | Использование внутреннего RAM (DMA-режим) |
| 2 | ACC_DIR2 | Авто-декремент счётчика `ACC_CNT` |
| 3 | ACC_DIR3 | Загрузка константы в `RGACC` |
| 4 | ACC_DIR4 | Использование `AGR[]` как адреса (вертикальный режим) |
| 5 | ACC_DIR5 | Инкремент адреса `AA` после каждого цикла |
| 6 | ACC_DIR6 | Двойной CAS + выбор IDE вместо DI для MDOY |
| 7 | — | Зарезервирован |

## Внутренние регистры

### RGACC[7:0] — регистр константы/данных

```vhdl
RGACC[].clk = /MR
RGACC[].ena = DFF((/M1 & /RF & ACC_DIR3), CLK_Z80)
RGACC[].d   = DI[]
```

Загружается из шины данных во время `M1+RF` если `ACC_DIR3=1`. В режиме заливки (Mode 1) содержит байт-заполнитель.

### ACC_CNT[7:0] — счётчик байт

```vhdl
CASE DFF(START_ACC, CLK42) IS
    WHEN 1 => ACC_CNT[].d = RGACC[]    -- загрузка начального значения
    WHEN 0 => ACC_CNT[].d = ACC_CNT[] - 1  -- декремент
END CASE;
```

Загружается значением `RGACC` при старте, затем уменьшается каждый цикл (если `ACC_DIR2=1`).

Завершение:
```vhdl
ACC_END.d = (ACC_CNT[] == 1) or !ACC_DIR2
```

Если `ACC_DIR2=0` — операция одного цикла. Если `ACC_DIR2=1` — продолжается до `ACC_CNT=1`.

### AA[15:0] — текущий адрес

```vhdl
AA[].ena = START_ACC or (ACC_DIR5 & !(CAS or START_ACC) & (ACC_GO or (ACC_GO_1 & ACC_DIR6)))

CASE DFF(START_ACC, CLK42) IS
    WHEN 1 => AA[].d = AI[]        -- захват адреса Z80 при старте
    WHEN 0 => AA[].d = AA[] + 1    -- инкремент после каждого цикла
END CASE;
```

`AO[15:0] = AA[15:0]` — подставляется вместо адреса Z80 в DCP.

### AGR[7:0] — строчный адрес (G_LINE)

```vhdl
CASE DFF(START_ACC, CLK42) IS
    WHEN 0 => AGR[].d = AGR[] + 1        -- авто-инкремент
    WHEN 1 => AGR[].d = DI[]             -- загрузка из шины
END CASE;
```

Отдельно обновляется при записи в порт `0xC4` (`DCP == 1100X100`). Выводится как `G_LINE[7:0]` в видеоконтроллер — содержит номер текущей строки кадра.

### XCNT/XAGR/AAGR — дополнительный адресный счётчик (ALT_ACC)

Используется при `ALT_ACC=1` (устанавливается записью в порт `0xC7`).

```vhdl
XCNT_AGR[15:0] = (XCNT[], XAGR[]) + (000000b, AAGR[])
CASE !DFF(START_ACC, CLK42) IS
    WHEN 1 => (XCNT[].d, XAGR[].d) = XCNT_AGR[]  -- инкремент на AAGR
    WHEN 0 => AAGR[].d  = (AI9,AI8, DI[])         -- загрузка шага
              (XCNT[], XAGR[]) = (00b, AI[15:10], 00000000b) -- загрузка базы
END CASE;
```

`AAGR` — шаг перемещения (задаётся из `AI[9:8]` и `DI`). `XCNT:XAGR` — 16-битный адрес с переносом. Это позволяет ускорителю работать с двумерными структурами памяти (например, видеостроки с шагом не равным 1).

### Внутренний RAM (256×16)

```vhdl
RAM_WR = DFF(!/RD & (!ACC_GO & CT0) & ACC_DIR1, CLK42)
RAM.wraddress[] = RAM_ADR[]  -- ACC_CNT или XCNT (зависит от ALT_ACC)
RAM.data[]      = (XMD[], XMD[])  -- записываем обработанный байт
```

При `ACC_DIR1=1` прочитанные байты сохраняются во внутренний RAM. Следующий цикл берёт данные уже из RAM, а не из DRAM. Это позволяет делать «copy+modify» без двойного обращения к DRAM.

## Логические операции (FN_ACC)

При обнаружении инструкций `AND/OR/XOR (HL)` ускоритель может применить логическую операцию к данным:

```vhdl
FN_ACC[].d = DI7 & !DI6 & !PRF_CMD & !(DI[5:3])
```

Это инструкции с кодами `0x86` (ADD/AND), `0x96` (SUB), `0xA6` (AND), `0xAE` (XOR), `0xB6` (OR), `0xBE` (CP):

```vhdl
CASE FN_ACC[1:0] IS
    WHEN 0 => XMD[] = MD[]               -- BE: просто данные
    WHEN 1 => XMD[] = MD[] or RAM.q[]    -- B6: OR с данными из RAM
    WHEN 2 => XMD[] = MD[] xor RAM.q[]   -- AE: XOR
    WHEN 3 => XMD[] = MD[] and RAM.q[]   -- A6: AND
END CASE;
```

`XMD[]` — модифицированные данные, которые запишутся обратно в RAM.

## Пути данных

### Источник MDO (16-битные данные для DRAM)

```vhdl
-- MDOX: младший байт
CASE MC_END & HDD_FLIP IS
    WHEN 0 => MDOX[] = DI[]     -- обычные данные CPU
    WHEN 1 => MDOX[] = HDDR[]   -- старший байт IDE
END CASE;

-- MDOY: для верхнего байта
CASE ACC_DIR6 IS
    WHEN 0 => MDOY[] = DI[]     -- обычный
    WHEN 1 => MDOY[] = HDDR[]   -- IDE DMA
END CASE;

-- Итоговый MDO
CASE /IO & ACC_DIR1 IS
    WHEN 0 => MDO[] = (MDOY[], MDOX[])     -- обычный путь
    WHEN 1 => MDO[] = (RAM.q[7:0], RAM.q[7:0])  -- из внутреннего RAM
END CASE;
```

### Сигнал GLISSER (заливка 0xFF)

```vhdl
CASE ACC_DIR1 IS
    WHEN 0 => GLISS_R = (DI[] == 0xFF)          -- байт от CPU
    WHEN 1 => GLISS_R = (RAM.q[7:4] == 0xF) & (RAM.q[3:0] == 0xF)  -- из RAM
END CASE;
GLISSER = GLISS_R
```

Сигнал `GLISSER` сообщает видеоконтроллеру, что текущий байт = `0xFF` (белый пиксель). Используется для оптимизации рисования.

## Управление циклами DCP

### START_ACC — запуск ускорителя

```vhdl
START_ACC = !/MR or !/M1 or !/RF or !ACC_BLK or !ACC_DIR0 or MC_TYPE
```

`START_ACC = 0` (активен) когда: идёт цикл выборки команды (`/MR & /M1 & /RF`), ускоритель разблокирован (`ACC_BLK`), режим ненулевой (`ACC_DIR0`), и цикл не запущен (`!MC_TYPE`).

### CONTINUE — разрешение завершения

```vhdl
CONTINUE = ACC_END
```

DCP ждёт `CONTINUE=1` перед завершением цикла. Ускоритель выдаёт `CONTINUE` через `ACC_END`, который устанавливается при `ACC_CNT=1` или при однократной операции.

### DOUBLE_CAS — удвоенный CAS

```vhdl
DOUBLE_CAS = ACC_DIR6 & !START_ACC
```

При `DOUBLE_CAS=1` в `SP2_ACEX.TDF` формируются удвоенные строб-импульсы CAS:
```vhdl
CASXE0 = (MCA0 == 0) or ((MCA0 == 1) & DOUBLE_CAS)
CASXE1 = (MCA0 == 1) or ((MCA0 == 0) & DOUBLE_CAS)
```

Это позволяет за один такт передать сразу два байта из DRAM (для 16-битного копирования).

## Взаимодействие с IDE

При `HDD_FLIP=1` (идёт второй байт 16-битного IDE-чтения):
- `MDOX[] = HDDR[]` — берётся буферизованный старший байт IDE
- При `ACC_DIR6=1` — `MDOY[] = HDDR[]` тоже

Это позволяет DMA-читать диск: CPU выполняет ускоренный `LD r,r`, ускоритель берёт данные прямо из регистра данных IDE и пишет их в указанную область памяти.

## Взаимодействие с Covox DMA

Covox DMA активируется когда в `SP2_ACEX.TDF`:
```vhdl
CBL_WR = ... or
    (DFF((DECODE.PAGE[7:0] == 0xFD), CLK42, (CBL_INT_ENA & ACC.ACC_DIR1)) &
     !DFF((DECODE.MC_WRITE or DECODE.CAS), CLK42, , /RESET))
```

При `ACC_DIR1=1` и `PAGE=0xFD` данные из `ACC.MDO[]` автоматически попадают в FIFO буфер Covox. Это позволяет без участия CPU заполнять звуковой буфер данными из памяти.

## Взаимодействие с видеоконтроллером

- `G_LINE[7:0]` → `SVIDEO.ZX_PORT[5:0]` — текущая строка кадра для ZX-адресации
- `GLISSER` через `SP2_ACEX` → `SVIDEO.wr` — принудительная запись в видеобуфер
- При `DOUBLE_CAS=1` → `SVIDEO.DOUBLE_CAS=1` — видеоконтроллер удваивает CAS-циклы

## ACC_BLK — блокировка при прерываниях

```vhdl
ACC_BLK.clk = /M1
ACC_BLK.d   = DFF((/IO & ACC_BLK) or (!ACC_BLK & RETI), CLK_Z80)
ACC_BLK.prn = /RESET & ACC_MODE3
```

`ACC_BLK` блокирует ускоритель при входе в обработчик прерывания (`ACC_MODE3=1`). Разблокируется командой `RETI` или `RETN`. При `ACC_MODE3=0` ускоритель работает и внутри обработчиков прерывания.

## Флаг CORRECT_1F

```vhdl
CORRECT_1F = IN_OUT_CMD & (DO[] == 0x1F) & !/MR & !/RD
DO[4:3].clrn = !CORRECT_1F
```

Специальный корректор: при чтении через `IN A,(C)` ускоритель сбрасывает биты 4–3 выходного байта если значение = `0x1F`. Это патч для совместимости с конкретными Z80-программами.

## Примеры использования

### Быстрая заливка области памяти

```asm
; Включить ускоритель
LD A, 1
OUT (0xC3), A       ; ALL_MODE0=1

; Установить константу и счётчик
LD A, 0xFF          ; байт-заполнитель
LD B, A             ; заливка: следующая LD C,C прочитает его из RGACC
                    ; RGACC заполняется при M1+RF с ACC_DIR3=1

LD BC, 256          ; B=количество байт, C=константа
LD DE, 0x4000       ; адрес назначения
LD (DE), A          ; LD C,C — режим 1 (заливка)
DEFB 0x49           ; LD C,C — запускает ускоритель режима 1
```

### Копирование блока памяти

```asm
LD HL, src          ; источник
LD DE, dst          ; назначение
LD BC, size         ; размер
DEFB 0x6D           ; LD L,L — режим 5 (копирование строки)
; ускоритель автоматически инкрементирует HL/DE и декрементирует BC
```

## Практические рекомендации

1. **Всегда проверяйте `ALL_MODE0=1`** перед использованием специальных опкодов. Иначе Z80 выполнит `LD r,r` как обычную NOP-инструкцию.

2. **Ускоритель не перехватывает I/O** (`/IO=0`). Работает только с обращениями к памяти.

3. **Прерывания и ускоритель:** если `ALL_MODE3=0`, прерывания клавиатуры отключаются при активном ускорителе. Если `ALL_MODE3=1` — прерывания разрешены, но `ACC_BLK` не блокирует ускоритель в ISR (только при `ACC_MODE3=1`).

4. **DOUBLE_CAS** требует, чтобы видеоконтроллер и DRAM были готовы к удвоенным импульсам. Не используйте `ACC_DIR6=1` при обращении к обычной RAM без проверки.

5. **ALT_ACC** (запись в порт `0xC7`) позволяет работать с двумерными структурами, задавая шаг строки через `AAGR`. Используется для побитового рисования спрайтов.
