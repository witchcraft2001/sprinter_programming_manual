## Что такое DCP

**DCP** (Dynamic Configuration Port) — это не просто "таблица портов", а
центральный маршрутизатор всех I/O-циклов Sprinter.

Он реализован в FPGA как **память 256 x 16 бит** (`DCP.MIF`/`DCP.TDF`) и на
каждый `IN`/`OUT` отвечает на два вопроса:

1. **Кто обслуживает этот порт?**
2. **Сколько wait states нужно вставить?**

В терминах железа цикл выглядит так:

1. младший байт номера порта `A[7..0]` используется как адрес в таблице DCP;
2. из таблицы читается 16-битный дескриптор;
3. поле `TYPE` выбирает класс устройства и задержки;
4. поле `MA[11..0]` передаёт конкретные управляющие биты во внутреннюю логику.

То есть DCP удобно понимать как **словарь "порт -> действие"**. Вместо
жёстко распаянного дешифратора Sprinter использует небольшую память, где для
каждого порта заранее записано, что именно он делает.

```
DCP.MIF: 256 × 16 бит
─────────────────────────
Addr (8 бит) = номер порта Z80

  bit 15─12   TYPE[3..0]   тип операции
  bit 14─12   wait state selector
  bit 11─ 0   MA[11..0]    управление MAX7000, HDD, CMOS
```

---

## Что лежит в 16-битной записи

```
бит 15..12   TYPE    класс устройства / тип I/O операции
бит 14..12   WSEL    выбор профиля wait states
бит 11..0    MA      управляющие биты для внутренних блоков
```

Поле `TYPE` определяет не конкретный "порт 1F0" или "порт 1C", а именно
семейство поведения. Например, "клавиатура", "мышь", "MAX7000/CMOS",
"системный порт".

## Поле TYPE[3..0]

| TYPE (биты 15–12) | Значение | Описание |
|-------------------|---------|----------|
| `0b0000` = `0` | Нет функции | Порт не декодирован, нет реакции |
| `0b0001` = `1` | HDD данные | Данные IDE (`INIR`/`OTIR`) |
| `0b0010` = `2` | Запись | Запись в системный регистр |
| `0b0011` = `3` | Чтение AY/COVOX | Возврат данных |
| `0b0100` = `4` | Клавиатура | Чтение клавиш |
| `0b0101` = `5` | Kempston Mouse | Чтение мыши |
| `0b0110` = `6` | Видео | Чтение видеорегистра |
| `0b0111` = `7` | MAX7000 | FDD/HDD регистры/CMOS через MAX7000 |
| `0b1100`–`0b1111` = `C–F` | Системные | Переключение страниц памяти, режимов |

## I/O Wait States из W_TAB

Таблица `W_TAB` в `DCP.TDF` задаёт **I/O wait states**, вставляемые при выполнении `IN`/`OUT`. Она индексируется младшими 3 битами TYPE-поля (`MEM.q[14..12]`) и состоянием `TURBO`.

> **Важно:** эти значения относятся только к I/O циклам. Для memory-циклов (выборка инструкций, `LD r,(HL)`) работает отдельный механизм `/MR_WAIT` — см. раздел [2.3 Кэш и wait states](03_cache.md).

```
CASE (TURBO, MEM.q[14..12]) IS
    Нормальный (TURBO=0)                Turbo (TURBO=1)
    TYPE=0 → W_TAB = 2                  TYPE=0 → W_TAB = 2
    TYPE=1 → W_TAB = 2  (default 1000)  TYPE=1 → W_TAB = 2
    TYPE=2 → W_TAB = 1  (write ports)   TYPE=2 → W_TAB = 4
    TYPE=3 → W_TAB = 1  (AY read)       TYPE=3 → W_TAB = 4
    TYPE=4 → W_TAB = 1  (kbd/system)    TYPE=4 → W_TAB = 7
    TYPE=5 → W_TAB = 2  (Kempston)      TYPE=5 → W_TAB = 7
    TYPE=6 → W_TAB = 7  (HDD CS/ZX)     TYPE=6 → W_TAB = 7
    TYPE=7 → W_TAB = 10 (MAX7000)       TYPE=7 → W_TAB = 10
END CASE;
```

**IDE данные — отдельный случай (HDD_W):**
```
MEM.q[11..8] == 0 → HDD_W =  4  (порт 1F0h — данные)
иначе              → HDD_W = 10  (регистры 1F1h–1F7h)
```

### Сводная таблица I/O waits

| Группа портов | Нормальный (7 МГц) | Turbo (21 МГц) |
|---------------|---------------------|-----------------|
| Default (`1000`) | 2 | 2 |
| Write: банки, AY, COVOX (`2000`) | 1 | **4** |
| AY read (`3000`) | 1 | **4** |
| Клавиатура, системные (`4000`, `C000`) | 1 | **7** |
| Kempston Mouse (`5000`) | 2 | **7** |
| HDD CS / ZX-видео (`6000`) | 7 | 7 |
| MAX7000: FDD, CMOS (`7000`) | 10 | 10 |
| IDE регистры | 10 | 10 |
| IDE данные | 4 | 4 |

> Memory access (`LD r,(HL)`, fetch кода) не подчиняется W_TAB: в нормальном 7 МГц режиме memory wait отсутствует, в турбо — активен до сигнала `MC_END` и снимается при попадании в кэш-область или при активном акселераторе.

## Содержимое DCP.MIF (ключевые записи)

Файл `DCP.MIF` — это исходная таблица декодирования. Формат: `адрес : значение`.

```
[0..FF] : 1000     ; по умолчанию — нет функции (TYPE=1, нет реакции)

0       : 1040     ; DCP PAGE (системный)

10      :          ; FDD / периферия через MAX7000
    7018  ; RD WG93 порт 1F/0F
    7019  ; RD WG93 порт 3F
    701A  ; RD WG93 порт 5F
    701B  ; RD WG93 порт 7F
    7017  ; WR_PDOS FF
    701F  ; RD_KEYS / WR_A20

1C      : 71D8     ; CMOS_DAT_RD
1D      : 70D4     ; CMOS_ADR_WR
1E      : 71D4     ; CMOS_DAT_WR

20      :          ; IDE CS1 и CS3 порты
    60A8  ; HD_CS1 (порт 1F0)
    62A8  ; HD_CS1 (порт 1F1)
    ...
    6DA8  ; HD_CS3 (порт 3F6)
    6FA8  ; HD_CS3 (порт 3F7)
    7060  ; Set HDD1
    7064  ; Set HDD2
    7120  ; Set 320 Lines
    7124  ; Set 312 Lines
    7160  ; Soft Reset

30      :          ; ISA слот
    7000  ; slot 1 ports
    7001  ; slot 2 ports
    7002  ; slot 1 mem
    7003  ; slot 2 mem

40      : 4000     ; клавиатура (TYPE=4)
52      : 3000     ; AY_D READ (TYPE=3)
58      : 5000     ; Kempston Mouse (TYPE=5)

[80..FF]: C000     ; Системные порты (TYPE=C), банкинг

88      : 2000     ; COVOX данные
89      : 2000     ; COVOX режим

8C      : 3000     ; AY_D READ
8D      : 2000     ; AY_A WRITE
8E      : 2000     ; AY_D WRITE
8F      : 2000     ; ROM_WRITE

90..9F  :          ; RAM PAGES (алиасы страниц)
    3030 3031 2032 2033 ...

B0..BF  :          ; RAM PAGES (алиасы)
C0..CF  : 2000     ; SYS PORTS (копии системных портов)
D0..DF  :          ; RAM PAGES
E0..EF  :          ; ROM PAGES
F0..FF  :          ; RAM PAGES
```

## Системные регистры (запись в диапазон #C0–#CF)

Запись в порты `#C0`–`#CF` управляет внутренними регистрами DCP. В DCP.TDF задекодировано через `DCPP[] == B"1100XXXX"`:

| DCPP-паттерн | Порт(ы) | Регистр |
|-------------|---------|---------|
| `B"1100X000"` | `#C0`, `#C8` | `SC[7..0]` — конфигурация режима |
| `B"1100X001"` | `#C1`, `#C9` | `PN[7..0]` — нумерация страниц |
| `B"1100X010"` | `#C2`, `#CA` | `BORDER` — бордюр |
| `B"1100X011"` | `#C3`, `#CB` | `ALL_MODE` — графический режим |
| `B"1100X101"` | `#C5`, `#CD` | `RGMOD` — double buffer, screen |
| `B"11001011"` | `#CB` | `HOLD` — акселератор |

### Регистр SC (System Configuration)

Запись в `#C0` (или `#C8`):

| Бит | Имя | Назначение |
|-----|-----|-----------|
| 0 | SC0 | ZX-режим: 1 = ZX Spectrum совместимость |
| 1 | SC1 | ROM16: 1 = 16-битная ROM |
| 2 | SC2 | NMI разрешение |
| 3 | SC3 | Дополнительный режим |
| 4 | SC4 | `BLK_R` — доступ к расширенной памяти |

```asm
; Включить нативный режим Sprinter (SC0=0)
        ld      a, SYS_PORT_OFF ; #3C
        out     (#3C), a

; Активировать расширенный блок памяти (SC4=1)
SYS_PORT_ON     EQU     #7C
        ld      a, SYS_PORT_ON
        out     (SYS_PORT_ON), a
```

### Регистр CNF (Configuration)

Управляет глобальными флагами:

| Бит | Имя | Назначение |
|-----|-----|-----------|
| 3 | CNF3 | Scorpion режим |
| 4 | CNF4 | ROM16 on/off |
| 5 | CNF5 | Сброс PN5 |
| 6 | CNF6 | Scorpion-OFF |
| 7 | CNF7 | Pentagon-512 |

## Как DCPP управляет работой ACEX

После начала I/O-цикла низкий байт порта защёлкивается в `DCPP[7..0]` и живёт
до конца этого цикла. Дальше уже не CPU "знает", что обращался к порту `#8D`
или `#1D`, а вся логика ACEX смотрит именно на `DCPP` и на считанную DCP-запись.

Поэтому один и тот же механизм одновременно управляет:

```
Видео-контроллер проверяет DCPP для RGMOD, BORDER, ALL_MODE
COVOX-блок реагирует на DCPP == #88 (данные) и #89 (режим)
AY-синтезатор смотрит на DCPP == #8D (адрес) и #8E (данные)
ROM_RG активируется при DCPP == #8F
ISA контроллер: DCPP == B"1111XXXX"
Банкинг: DCPP == B"1100XXXX" → запись в SC/PN/BORDER...
```

Именно это делает DCP фундаментом всей архитектуры Sprinter: клавиатура,
мышь, CMOS, IDE, страницы памяти и системные режимы все проходят через один
декодер.

## Динамическое перепрограммирование DCP

Здесь важно разделять два уровня.

### Что подтверждено текущими FPGA-исходниками

Подтверждено следующее:

- DCP — это **настоящая RAM-таблица**, а не только статически прошитая логика;
- в `DCP.TDF` существует аппаратный путь записи в неё:
  - `MEM.wren = MEM_WR`
  - `MEM.wraddress = PGS[]`
  - `MEM.data = MEM_D[]`
- в `DCP.MIF` запись `0 : 1040` помечена как **`DCP PAGE`**.

Это означает, что архитектурно Sprinter действительно поддерживает
**динамическое изменение дешифратора портов**, а не только компиляцию таблицы в
битстрим.

### Что можно считать штатным API

Старые документы описывают экспериментальные функции `DCP_FN0`, `SET_PORTS`,
`READ_PORTS`, `WRITE_PORTS` (`#F4`, `#F8`, `#F9`, `#FA`), а также идею
`user ports` в диапазоне `#80..#BF`. Но в этих же документах они прямо
помечены как **"не доделана" / "в разработке"**, а в текущих BIOS-исходниках
нет подтверждённого публичного интерфейса, который можно считать переносимым
стандартом.

Поэтому для manual корректно фиксировать только следующее:

1. **аппаратная возможность перепрограммирования DCP существует;**
2. **штатно документирован и безопасен прежде всего доступ через обычные
   системные порты и предопределённую таблицу `DCP.MIF`;**
3. **публичный переносимый API для произвольной перенастройки DCP в актуальных
   исходниках не подтверждён.**

## Практические выводы для программиста

1. **Порт = функция**: если порт есть в DCP.MIF с ненулевым TYPE — он работает; иначе — нет реакции.
2. **I/O wait states зависят от TYPE**: write-порты быстрее, FDD/MAX7000 — в разы медленнее. Memory access управляется отдельно через `/MR_WAIT` и кэш.
3. **Системные порты `#C0`–`#CF`**: управляют режимами чипа — не используйте их случайно.
4. **IDE**: запись и чтение — разные физические порты (`HDW_xxx` vs `HDR_xxx`).
5. **ISA**: адрес порта формируется через `ISA_PORT` регистр, не напрямую.
6. **DCP удобнее мыслить как маршрутизатор I/O**, а не как список "магических портов". Это помогает понимать, почему у разных групп портов разные задержки и разные побочные эффекты.

---

## Ключевые моменты

> - DCP — 256×16-бит таблица-маршрутизатор в FPGA: каждый I/O-цикл сначала проходит через неё.
> - I/O wait states из W_TAB: write-порты 1/4 такта (norm/turbo), клавиатура/система 1/7, Kempston 2/7, HDD CS 7/7, MAX7000 10/10; IDE данные — отдельно 4 такта.
> - Memory access не зависит от W_TAB: в 7 МГц нет доп. waits, в турбо — динамические waits, снимаемые кэшем или акселератором.
> - Диапазон `#C0`–`#CF` — системные регистры (SC, PN, BORDER, ALL_MODE, RGMOD).
> - Аппаратная запись в DCP подтверждена, но переносимый публичный API для свободной перепрограммировки в текущих исходниках не подтверждён.
