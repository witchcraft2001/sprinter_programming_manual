# 5.7 Двойная буферизация (Double Buffering)

> **Навигация:** [← 5.6 Акселератор](06_accelerator.md) | [Оглавление](../README.md) | [5.8 Текстовый режим →](08_text_mode.md)

---

## Зачем нужна двойная буферизация

Без двойной буферизации программа рисует непосредственно в активный видеобуфер. Если обновление кадра занимает дольше одного интервала VSync, пользователь видит **tearing** — горизонтальные разрывы, где часть экрана уже обновлена, а часть ещё нет.

**Double buffering** решает проблему: программа рисует в **неактивный** (невидимый) буфер, затем в момент VSync переключает его на **активный**. Переключение мгновенное, tearing исчезает.

---

## Организация VRAM в Sprinter

VRAM 512 КБ позволяет хранить **два экрана 320×256** одновременно:

| Экран | Страницы VRAM | Размер |
|-------|--------------|--------|
| A (буфер 0) | `#50`, `#51`, `#52`, `#53`, `#54` | 80 КБ |
| B (буфер 1) | `#55`, `#56`, `#57`, `#58`, `#59` | 80 КБ |

Остальные страницы VRAM (`#5A`–`#5F`) используются под спрайтовые банки и вторичный текстовый экран.

---

## Регистр RGMOD (порт #C9)

Управляет режимом видеовывода. Из `SP2_ACEX.TDF` биты регистра подаются в видеоконтроллер напрямую:

```
SVIDEO.DIR_PORT[3] = RGMOD0;        -- screen page select
SVIDEO.DIR_PORT[4] = RGMOD1;        -- 1 = screen off, 0 = screen on
```

| Бит | Назначение |
|-----|-----------|
| 0 | **Активный буфер (double buffer flip):** 0 = экран A, 1 = экран B |
| 1 | **Screen on/off:** 0 = экран включён, 1 = экран выключен (если прошивка собрана с `SCREEN_OFF == "USE"`) |
| [7:2] | Не используются |

Практически программу интересуют только биты 0 и 1: бит 0 — переключение буфера в цикле отрисовки; бит 1 — возможность спрятать экран (например, для плавной загрузки картинки).

---

## Простая техника: flip

```asm
RGMOD           EQU #C9

; Переключить активный буфер
FlipBuffer:
        in      a, (RGMOD)
        xor     1                   ; инвертировать бит 0
        out     (RGMOD), a
        ret
```

Этот код мгновенно меняет активный экран. Если он выполняется **до VSync**, видимого tearing не будет.

---

## Правильный цикл с ожиданием VSync

```asm
GameLoop:
        ; 1. Определить, какой буфер сейчас активен
        in      a, (RGMOD)
        and     1                   ; A = 0 (A активен) или 1 (B активен)

        ; 2. Рисовать в противоположный буфер
        jr      z, .draw_to_b

.draw_to_a:
        ; Подключить страницы экрана A в окна
        ld      a, #50
        out     (#E2), a
        jr      .do_draw

.draw_to_b:
        ld      a, #55
        out     (#E2), a

.do_draw:
        ; ... отрисовка кадра ...
        call    DrawFrame

        ; 3. Ждать VSync
        halt                        ; ждать прерывание кадра (IM1)

        ; 4. Flip (переключить активный буфер)
        in      a, (RGMOD)
        xor     1
        out     (RGMOD), a

        jr      GameLoop
```

## Использование SDK: _screenActive

SDK из `zx-sprinter-sdk/sdk/src/sprinter/lib_startup.asm` хранит индекс активного экрана в RAM:

```asm
_screenActive:  db  0           ; 0 или 1

; Функция swap_screen
_swap_screen:
        ; ... ожидание VSync, отрисовка спрайтов ...

        call    update_pal

        call    waitVsync       ; halt

        ld      a, (_screenActive)
        out     (RGMOD), a
        xor     1
        ld      (_screenActive), a

        ; ... обновление спрайтов для нового буфера ...
        ret
```

## Пример из flappybird: ChangeVideoPage

```asm
ChangeVideoPage:
        in      a, (RGMOD)
        xor     1
        out     (RGMOD), a
        ret
```

Простейшая версия без ожидания VSync — подходит для нетребовательных игр.

## Координация с отрисовкой

Частая ошибка: рисовать и в A, и в B одновременно, что приводит к "double flashing". Правильно:

1. Считать `_screenActive` → определить "скрытый" буфер.
2. Рисовать весь кадр в скрытый буфер.
3. Flip.
4. Следующий кадр — теперь рисовать в *противоположный* буфер (бывший активный).

Это значит, что **каждый кадр рисуется заново** — нельзя "накапливать" изменения, потому что между кадрами активные буферы меняются.

### Оптимизация: "дельта"-обновления

Для статичного фона можно хранить его копию в RAM и быстро копировать при каждом кадре только изменённые области:

```asm
; Каждый кадр:
; 1. В скрытый буфер скопировать фон из RAM (через акселератор)
; 2. Нарисовать подвижные объекты
; 3. Flip
```

## Синхронизация с VSync

Прерывание кадра генерируется аппаратно в конце обратного хода. Частота 50 Гц (PAL) или 60 Гц (NTSC).

```asm
; Ждать следующего VSync
WaitVSync:
        halt                        ; остановить CPU до следующего прерывания
        ret
```

`HALT` работает только при разрешённых прерываниях (`EI`). В IM1 инструкция `RST #38` вызовется автоматически. В IM2 вектор зависит от установки.

---

## Частые ошибки

### Ошибка 1: flip без ожидания VSync

```asm
; ПЛОХО
        call    DrawFrame
        call    FlipBuffer          ; может произойти в середине кадра → tearing
```

### Ошибка 2: рисование в активный буфер

```asm
; ПЛОХО
        call    DrawFrame           ; без определения активного буфера
        ; видно промежуточное состояние отрисовки
```

### Ошибка 3: не учитывать смену активного буфера между кадрами

```asm
; ПЛОХО — каждый кадр одно и то же
.game_loop:
        call    DrawSpriteInA       ; всегда в A
        call    FlipBuffer
        jr      .game_loop
; Результат: в B ничего нет → мерцание
```

### Правильно

```asm
.game_loop:
        ; Определить скрытый буфер
        in      a, (RGMOD)
        and     1
        call    SetupHiddenBufferPages  ; подключить нужные VRAM страницы

        call    DrawFrame               ; полная отрисовка

        halt                            ; VSync

        ; Flip
        in      a, (RGMOD)
        xor     1
        out     (RGMOD), a

        jr      .game_loop
```

## Double buffering и палитра

Палитра **общая** для обоих буферов. Изменение цвета мгновенно отражается на активном экране. Это можно использовать для эффектов (фейд, цветовой циклинг).

---

## Ограничения

- Double buffering работает **только в режиме 320×256 8bpp** (в 640×512 нет места для второго буфера).
- В ZX Spectrum режиме double buffering недоступен (только один экран ZX).
- Требует прямой доступ к VRAM через окна — не работает через BIOS функции печати текста (они пишут в текстовый слой, не в графический буфер).

---

## Ключевые моменты

> - VRAM в Sprinter имеет место на 2 экрана 320×256 (A и B).
> - Активный буфер выбирается битом 0 порта `#C9` (RGMOD).
> - Цикл: определить скрытый буфер → рисовать → HALT (ждать VSync) → flip.
> - Каждый кадр рисуется полностью с нуля (нет инкрементального обновления).
> - Палитра общая для обоих буферов.
> - Double buffering работает только в режиме 320×256.
