# 5.4 Режим 640×256, 4bpp

> **Навигация:** [← 5.3 Режим 320×256](03_mode1_320.md) | [Оглавление](../README.md) | [5.5 Палитра →](05_palette.md)

---

## Характеристики

| Параметр | Значение |
|---------|---------|
| Разрешение | 640 × 256 пикселей |
| Биты на пиксель | 4 (16 цветов) |
| Формат | 2 пикселя на 1 байт (старший ниббл = левый пиксель) |
| Размер экрана | 640 × 256 / 2 = **81 920 байт** (~5 страниц по 16 КБ) |
| Страниц VRAM 16 КБ | 5 страниц (`#50`–`#54` — один экран) |
| Double buffering | Возможен (2 экрана по ~80 КБ помещаются в 512 КБ VRAM) |

Этот режим предназначен для **CAD-приложений, утилит и графиков** с высокой горизонтальной чёткостью (640 пикселей при 16 цветах). Размер экрана совпадает с режимом 320×256 8bpp.

---

## Включение режима

### Через DSS API

```asm
        ld      a, 2                ; 2 = 640×256 4bpp
        ld      b, 0
        ld      c, #50              ; SetVMod
        rst     #10
```

### Через прямые порты

```asm
        ld      a, 2
        out     (#C3), a            ; ALL_MODE = 640×256
```

---

## Формат пикселя

```
Байт: [Pixel_Left][Pixel_Right]
       7 6 5 4    3 2 1 0

   Старший ниббл — левый пиксель (индекс 0..15)
   Младший ниббл — правый пиксель
```

### Вычисление адреса

```
Адрес = BASE + (Y * 640 + X) / 2
Ниббл: (X mod 2 == 0) → старший (левый), иначе младший (правый)
```

---

## Установка пикселя

```asm
; Установить пиксель (X, Y) цвета A (0..15)
; Вход: DE = X, HL = Y, A = цвет (0..15)
SetPixel640:
        push    af

        ; Вычислить линейный адрес: Y * 640 + X
        ; Y * 640 = Y * 512 + Y * 128 = Y << 9 + Y << 7

        ld      h, 0
        add     hl, hl              ; Y * 2
        ; ... (здесь необходима 24-битная арифметика т.к. буфер > 64 КБ)

        ; Для простоты: расчёт через 16-битное переполнение с использованием
        ; прибавления строк по 320 байт (640/2)

        ; ... упрощённый вариант для небольших областей ...

        pop     af
        ret
```

**Замечание:** буфер 80 КБ (5 страниц по 16 КБ) не помещается в одно окно Z80. Для произвольного доступа требуется **переключение страниц VRAM** через порты банкинга.

---

## Работа с буфером > 64 КБ

Для доступа к любому пикселю нужно:
1. Вычислить **линейный 18-битный адрес**.
2. Определить страницу VRAM: `page = #50 + (linear / 16384)`.
3. Определить смещение в окне: `offset = linear mod 16384`.
4. Подключить страницу в окно (обычно WIN3 = `0xC000`).
5. Обратиться по адресу `0xC000 + offset`.

```asm
; Установить пиксель (X, Y) в режиме 640×256
; Вход: DE = X (0..639), BC = Y (0..511), A = цвет (0..15)

SetPixel640:
        push    af
        ; Вычислить Y*320 в 18-битной арифметике
        ; 320 = #140

        ; HL = Y * 320
        ld      h, b
        ld      l, c
        push    hl                  ; сохранить Y
        ld      a, h
        ld      h, 0                ; расширенный H
        add     hl, hl              ; Y * 2
        add     hl, hl              ; * 4
        add     hl, hl              ; * 8
        add     hl, hl              ; * 16
        add     hl, hl              ; * 32
        add     hl, hl              ; * 64
        ; HL = Y * 64 (потенциальное переполнение)
        pop     bc                  ; BC = исходный Y
        ; ... полная 24-битная реализация опущена для краткости
        ; Используйте 24-бит через пару регистров + переполнение

        pop     af
        ret
```

> **Практика:** для работы в 640×256 обычно используют **построчный буфер** в RAM, который затем выгружается построчно в VRAM через acceлератор. Прямой поэпиксельный доступ слишком медленный.

## Работа с цветом в 4bpp

В 4bpp цвет символа кодируется двумя нибблами: INK (биты 3–0) и PAPER (биты 7–4). Для заливки области одним цветом нужно **продублировать ниббл** в оба полубайта:

```asm
; Подготовить байт заливки цветом C (0..15)
; Результат: A = CC (оба ниббла одинаковые)
MakeFillByte:
        ld      a, c
        and     #0F             ; оставить нижние 4 бита
        ld      b, a
        rlca : rlca : rlca : rlca   ; сдвинуть в старший ниббл
        or      b               ; A = CCCC_CCCC
        ret
```

Пример заливочных байт:

| Цвет | Значение | Вид |
|------|---------|-----|
| 0 (чёрный) | `#00` | пустота |
| 7 (серый) | `#77` | тени, 3D-рамки |
| 8 (тёмно-серый) | `#88` | фон панели |
| 15 (белый) | `#FF` | яркий фон |

## Горизонтальная линия (пример из FlexNavigator)

FlexNavigator рисует горизонтальные линии для окон и рамок через акселератор. Из `fnwin.z80` — `PutHorizLine`:

```asm
YPORT           EQU #89
PAGE1           EQU #A2
VPAGE_TILES     EQU #50

; Нарисовать горизонтальную линию
; Вход: HL = адрес в VRAM (X/2 + BASE), A = Y строка, E = байт заливки, BC = длина (байт)
HorizLine:
        di
        in      a, (PAGE1)
        push    af
        ld      a, VPAGE_TILES
        out     (PAGE1), a          ; WIN1 = VRAM

        ld      a, (line_y)
        out     (YPORT), a          ; установить строку

        ; Заполнить через акселератор
        ld      d, d                ; установить размер блока
        ld      a, c                ; длина (байт) в A
        ld      b, b                ; сброс акселератора (перед заполнением)
        ld      c, c                ; LD C,C — заполнить
        ld      (hl), e             ; шаблон заливки
        ld      b, b                ; выключить акселератор

        ld      a, #C0
        out     (YPORT), a          ; PORT_Y в безопасную зону

        pop     af
        out     (PAGE1), a
        ei
        ret

line_y: db 0
```

## Рендеринг текста в 4bpp (пример из FlexNavigator)

FlexNavigator рисует символы собственным рендерером, не через DSS/BIOS API. Из `fnprint.z80` — подготовка цветовых буферов и отрисовка:

```asm
; Подготовка цветового буфера для символа
; Вход: B = атрибут (PAPER[7:4] | INK[3:0])
; Подготавливает два 8-байтных буфера: PrnINKp и PrnPAPp

InsPrintColor:
        ; INK: нижние 4 бита → продублировать в оба ниббла → инвертировать
        ld      a, b
        and     #0F              ; INK
        ld      c, a
        rlca : rlca : rlca : rlca
        or      c                ; A = INK | INK << 4
        cpl                      ; инвертировать
        ld      hl, PrnINKp
        ld      b, 8
.fill_ink:
        ld      (hl), a
        inc     hl
        djnz    .fill_ink

        ; PAPER: верхние 4 бита → продублировать → XOR с INK
        ld      a, b
        and     #F0
        ld      c, a
        rrca : rrca : rrca : rrca
        or      c                ; A = PAPER | PAPER << 4
        ; XOR с инвертированным INK...
        ret

PrnINKp:    ds 8
PrnPAPp:    ds 8
```

Собственно отрисовка символа на каждую строку шрифта (8 строк):

```asm
; Для каждой строки символа (8 пикселей высотой):
;   1. Загрузить байт шрифта (8 бит = 8 пикселей)
;   2. OUT (YPORT), Y — установить строку
;   3. result = (font_byte OR PAPER) XOR INK
;   4. Записать 4 байта результата в VRAM через акселератор (LD L,L)

        ld      a, VPAGE_TILES
        out     (PAGE1), a
.font_line:
        ld      l, l                 ; LD L,L — копировать через акселератор
        ld      a, (hl)              ; байт шрифта
        ld      b, b                 ; сброс
        ; ... обработка с INK/PAPER буферами ...
        or      (hl)                 ; OR с PAPER
        xor     (hl)                 ; XOR с INK
        ld      a, a                 ; LD A,A — вертикальное копирование
        ld      (bc), a              ; запись в VRAM
        ld      b, b
```

**Формула цвета символа:** `pixel = (font_bitmap OR PAPER_pattern) XOR INK_pattern`.

## Адресация: X пикселей → адрес в памяти

В 4bpp каждый байт содержит 2 пикселя. Преобразование координаты:

```asm
; X (пиксель) → адрес байта
; HL = X в пикселях
; Результат: HL = X / 2 (адрес байта), CF = нечётный пиксель (младший ниббл)

PixelToAddr:
        srl     h
        rr      l               ; HL = X / 2
        ; CF = 1 если X нечётное (= младший ниббл)
        ret

; Базовый адрес в VRAM: SET 6, H — включить бит 14 (диапазон #4000 при WIN1)
        set     6, h            ; HL → #4000..#7FFF (внутри WIN1)
```

Y-координата устанавливается через `OUT (YPORT), A` — это и есть номер строки (0..255).

## Ограничения режима

1. **Только 16 одновременных цветов** — каждый пиксель задаётся 4 битами (индексы 0–15 в палитре). Сами 16 цветов выбираются из полной палитры 16 777 216 (6-6-6 RGB), поэтому можно настроить любые 16 оттенков. Для фотореалистичной графики этого мало — используйте 320×256 8bpp (256 цветов).
2. **Упаковка 2 пикселя в байт** — сложнее работать с отдельными пикселями, чем в 8bpp.
3. **Прямой доступ к буферу** требует переключения VRAM-страниц (5 страниц на экран).
4. **Акселератор работает** с побайтовым заполнением/копированием, но не с отдельными пикселями-нибблами.

---

## Когда использовать 640×256

- **GUI-приложения** с высоким горизонтальным разрешением (файловые менеджеры, редакторы форм).
- **CAD/диаграммы** — тонкие линии при 640 пикселях выглядят чётко.
- **Режим с собственным текстовым рендерером** (переменная ширина, антиалиасинг, нестандартные шрифты).
- **Утилиты**, где 16 цветов достаточно.

Для **простого текстового вывода** без собственного рендерера используйте [текстовый режим 80×32](08_text_mode.md) через DSS/BIOS API.

Для **игр** предпочтительнее 320×256 8bpp: 256 цветов, double buffering, 1 байт = 1 пиксель.

---

## Ключевые моменты

> - 640×256 4bpp, 2 пикселя на байт, старший ниббл = левый пиксель.
> - Буфер ~80 КБ = 5 страниц VRAM (`#50`–`#54`).
> - Double buffering возможен (экран ~80 КБ, как и 320×256).
> - X-адресация: `byte_addr = X / 2`; Y-координата через `OUT (YPORT), A`.
> - Дублирование ниббла для заливки: `RLCA×4 : OR original`.
> - Формула рендеринга символа: `(font OR PAPER) XOR INK`.
> - FlexNavigator — практический пример полноценного GUI в этом режиме.
> - Для простого текста — DSS text API ([5.8 Текстовый режим](08_text_mode.md)).
