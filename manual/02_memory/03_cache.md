## Что такое "кэш" в Sprinter

Sprinter не имеет **кэша** в привычном x86/ARM смысле (L1/L2 буфер на той же кристалле CPU). Вместо этого, термин *cache* в исходниках FPGA (`CASH_ON`, `CS_CASH`) обозначает **механизм ускорения обращений к определённым зонам памяти**. Практически это даёт то же, что и аппаратный кэш: часть адресного пространства начинает отвечать CPU без wait states.

Механизм нужен в первую очередь **для турбо-режима (21 МГц)**. При 7 МГц Z80 DRAM и Flash «успевают» за процессором и почти не требуют дополнительных тактов ожидания. А вот в турбо (21 МГц) та же самая DRAM физически не может отдать байт за 1 такт CLK_Z80 — поэтому без кэша каждое обращение к памяти получает вставку `/WAIT`.

### Целевое применение

- **Ускорение горячего блока кода.** Перед запуском критичного цикла (распаковка, обработка сэмпла, DMA-эмуляция) программа активирует кэш-область, и цикл выполняется на полной скорости турбо.
- **Ускорение вывода на экран.** Блочное копирование из рабочего буфера в VRAM при активном кэше идёт существенно быстрее.
- **Ускоренный доступ к резидентным страницам DSS** — когда DSS/BIOS обращается к своим данным.

### Управление кэшем

В FPGA сигнал `CASH_ON` управляется чтением порта `#88` с определённым битом адреса (см. `SP2_ACEX.TDF`):

```
CASH_ON = DFFE(A7, (/IO or /RD), /RESET,, DFF((A[7..0] == B"X1111011"), CLK42,,));
```

> **Для прикладного программиста:** почти всегда кэшем управляет BIOS / DSS / SDK startup-код, и заботиться о нём вручную не нужно. Прямое управление `CASH_ON` имеет смысл только в загрузчиках, системном коде или при микрооптимизации критических циклов. Неправильное обращение с кэшем может привести к чтению «несвежих» данных.

## Wait states — две независимые системы

В Sprinter два независимых механизма вставки тактов ожидания:

1. **I/O wait (`/IO_WAIT`)** — применяется при `IN`/`OUT` (порты ввода-вывода). Число тактов берётся из таблицы `W_TAB` и зависит от TYPE-поля DCP и от режима турбо.
2. **Memory wait (`/MR_WAIT`)** — применяется при чтении/записи памяти (`LD A,(HL)`, выборка инструкций и т. п.). Зависит от того, включён ли турбо, активен ли акселератор, попадает ли обращение в кэш-область.

Оба сигнала объединяются в общий `/WAIT` Z80:

```
/WAIT = /IO_WAIT & /MR_WAIT
```

## I/O wait states (W_TAB из DCP.TDF)

Таблица `W_TAB` из `DCP.TDF` индексируется битами `MEM.q[14..12]` (младшие 3 бита TYPE-поля ответа DCP) и состоянием `TURBO`:

```
CASE (TURBO, MEM.q[14..12]) IS
    -- TURBO = 0 (7 МГц)               -- TURBO = 1 (21 МГц)
    WHEN 0 => W_TAB[] = 2;  WHEN  8 => W_TAB[] = 2;
    WHEN 1 => W_TAB[] = 2;  WHEN  9 => W_TAB[] = 2;
    WHEN 2 => W_TAB[] = 1;  WHEN 10 => W_TAB[] = 4;
    WHEN 3 => W_TAB[] = 1;  WHEN 11 => W_TAB[] = 4;
    WHEN 4 => W_TAB[] = 1;  WHEN 12 => W_TAB[] = 7;
    WHEN 5 => W_TAB[] = 2;  WHEN 13 => W_TAB[] = 7;
    WHEN 6 => W_TAB[] = 7;  WHEN 14 => W_TAB[] = 7;
    WHEN 7 => W_TAB[] = 10; WHEN 15 => W_TAB[] = 10;
END CASE;
```

### Нормальный режим (7 МГц)

| MEM.q[14..12] | Waits | Примеры портов (DCP.MIF) |
|---------------|-------|--------------------------|
| 0 | 2 | — (не назначен) |
| 1 | 2 | `1000` — незанятые порты (default) |
| 2 | 1 | `2000` — запись: банки RAM `#82`/`#A2`/…, AY `#8D`/`#8E`, COVOX `#88`/`#89`, ROM `#8F` |
| 3 | 1 | `3000` — чтение AY (`#52`, `#8C`) |
| 4 | 1 | `4000` — чтение клавиатуры, `C000` — системные порты (`#C0`–`#CF`) |
| 5 | 2 | `5000` — Kempston Mouse (`#58`) |
| 6 | 7 | `6000` — HDD CS / ZX-видео |
| 7 | 10 | `7000` — MAX7000 (FDD, CMOS, HDD switches) |

### Турбо-режим (21 МГц)

| MEM.q[14..12] | Waits | Комментарий |
|---------------|-------|-------------|
| 0 | 2 | — |
| 1 | 2 | default |
| 2 | **4** | банки, AY/COVOX запись — заметно медленнее |
| 3 | **4** | AY read |
| 4 | **7** | клавиатура, системные порты |
| 5 | **7** | Kempston Mouse |
| 6 | 7 | HDD CS / ZX-видео |
| 7 | 10 | MAX7000 |

### Дополнительно: HDD данные

Для портов с HDD-данными работает отдельная таблица:

```
CASE LCELL(MEM.q[11..8] == 0) IS
    WHEN 0 => HDD_W[] = 10;   -- HDD регистры (1F1h–1F7h)
    WHEN 1 => HDD_W[] =  4;   -- HDD данные (1F0h)
END CASE;
```

То есть данные IDE читаются в 2.5 раза быстрее регистров — оптимизация для `INIR`/`OTIR` блочной передачи сектора.

> **Важно:** эти значения — это именно **I/O waits**, вставляемые в цикл `IN`/`OUT`. Они не относятся напрямую к чтению кода из ROM или к обращениям в DRAM — те проходят через отдельный механизм `/MR_WAIT`.

## Memory wait states (/MR_WAIT)

Обращения к памяти (выборка инструкций, `LD r,(HL)` и т. д.) управляются отдельно:

```
/MR_WAIT = LCELL(MC_END or MEM_RW or /CASH or (!TURBO & !ACC_ON));
```

Разберём эту формулу:

- **`MC_END`** — сигнал «цикл памяти завершён». Пока DRAM-контроллер формирует RAS/CAS, `MC_END=0` и `/MR_WAIT` активен.
- **`MEM_RW`** — идёт обновление (refresh) DRAM; на это время CPU тоже в wait.
- **`/CASH`** — если адрес попал в кэш-область, wait сразу снимается.
- **`(!TURBO & !ACC_ON)`** — **в обычном 7 МГц режиме без акселератора этот терм всегда равен 1**, и wait снимается.

### Что это означает на практике

| Ситуация | Memory wait |
|---------|-------------|
| 7 МГц, код и данные в DRAM | **≈ 0 доп. тактов** (DRAM успевает за Z80) |
| 7 МГц, чтение из ROM/Flash | ~0 доп. тактов (Flash 70 нс успевает за циклом M1 = ~570 нс) |
| 21 МГц, горячий код в DRAM | Wait пока `MC_END` не даст сигнал готовности (несколько CLK42) |
| 21 МГц, DRAM в кэш-области (`/CASH=0`) | ≈ 0 доп. тактов — полная скорость |
| 21 МГц, чтение ROM/Flash | Flash не успевает за 21 МГц — вставляются waits до готовности |
| 21 МГц, акселератор активен (`ACC_ON=1`) | Акселератор сам управляет памятью, Z80 ждёт `CONTINUE` от акселератора |

Точное количество wait-тактов зависит от RAS/CAS таймингов DRAM-чипа и от текущего состояния DRAM-контроллера (конец refresh и т. д.). Поэтому дать одно «число waits для ROM в турбо» нельзя — это динамическая величина.

## Оптимизация производительности

### 1. Избегать обращений к VRAM в критическом коде

```asm
; ПЛОХО: частые записи в VRAM при двойном доступе CPU+VIDEO
loop:   ld      a, (VRAM_addr)
        inc     a
        ld      (VRAM_addr), a
        djnz    loop

; ЛУЧШЕ: работать в RAM, блочно копировать в VRAM через акселератор
```

### 2. Держать горячий код в DRAM

В турбо-режиме выполнение кода из DRAM при активном кэше — самое быстрое. Если горячий цикл вызывает BIOS/DSS (ROM или системные страницы), это резко замедляет работу:

```asm
; ПЛОХО в узком цикле
.loop:
        ld      c, #82                  ; BIOS LP_PRINT_SYM
        rst     #08                     ; обращение к ROM через RST
        djnz    .loop

; ЛУЧШЕ: своя функция в RAM
.loop:
        call    MyPrintChar             ; код и данные в DRAM
        djnz    .loop
```

### 3. Использовать аппаратный акселератор для блочных операций

Все блочные операции с VRAM (очистка, копирование, спрайты) в разы быстрее через `LD r,r` команды акселератора, чем через `LDIR`. См. раздел [5.6 Акселератор](../05_graphics/06_accelerator.md).

### 4. Оптимизировать I/O циклы

При работе с FDD через MAX7000 (10 waits на каждое `OUT`) массовые обращения сильно замедляют код. Там, где возможно, используйте IDE с блочными `INIR`/`OTIR` (4 waits на байт данных вместо 10 на каждый регистр).

## Типичные тайминги инструкций

На 7 МГц без доп. wait states (идеальный случай — DRAM, попадание в кэш / режим без турбо):

| Инструкция | T-states | Время, мкс (7 МГц) | Время, мкс (21 МГц) |
|-----------|----------|--------------------|---------------------|
| `NOP` | 4 | 0.57 | 0.19 |
| `LD r, r` | 4 | 0.57 | 0.19 |
| `LD r, (HL)` (RAM) | 7 | 1.00 | 0.33 |
| `JP nn` | 10 | 1.43 | 0.48 |
| `CALL nn` | 17 | 2.43 | 0.81 |
| `LDIR` (на байт) | 21 | 3.00 | 1.00 |

В турбо на практике инструкции замедляются wait-ами, и реальная производительность ближе к ~2× (а не 3×) в зависимости от того, откуда идёт код и в каком состоянии кэш. Для блочных операций с VRAM используйте акселератор — он работает на CLK42 = 42 МГц независимо от режима Z80.

### Пример: OUT в MAX7000

Инструкция `OUT (port), A` сама по себе — 11 T-states. Плюс I/O wait по `W_TAB`:

```
На 7 МГц, порт MAX7000 (TYPE=7, 10 waits):
    11 + 10 × (CLK21 cycle) ≈ 31 T-state ≈ 4.4 мкс
```

Для FDD это накладывает реальное ограничение: массовый опрос WG93-регистров через MAX7000 нельзя ускорить программно.

## Видеоконфликты в ZX-режиме

В **ZX Spectrum-совместимом режиме** (`SC0=1`) видеоконтроллер и Z80 конкурируют за доступ к странице 5 (ZX-экран, `0x4000`–`0x7FFF`):

- Когда Z80 обращается к этой странице в момент, когда видеоконтроллер читает ту же память, аппаратура вставляет wait до конца видео-транзакции.
- Поведение идентично оригинальному ZX Spectrum 48K (contended memory) и именно на этом построены классические multicolour-эффекты.

В нативном Sprinter-режиме (`SC0=0`) видеоконфликтов нет: VRAM и DRAM — разные физические чипы.

## Ключевые моменты

> - Wait states делятся на два независимых механизма: `/IO_WAIT` (для `IN`/`OUT`) и `/MR_WAIT` (для памяти).
> - I/O waits из `W_TAB` в турбо увеличиваются для большинства портов (RAM-банки, AY: 1→4; клавиатура/система: 1→7).
> - HDD данные (1F0h) читаются с 4 waits, HDD регистры — с 10 waits — `INIR`/`OTIR` значительно быстрее построкового опроса.
> - Memory wait в 7 МГц обычно ≈ 0: DRAM и Flash успевают за Z80 без кэша.
> - В 21 МГц memory wait динамический; кэш и акселератор снимают его для DRAM-областей, ROM/Flash всё равно медленнее.
> - «Кэш» в Sprinter — механизм быстрого доступа, управляется BIOS/DSS/startup-кодом; прикладным программам вручную он обычно не нужен.
> - Акселератор работает на CLK42 и не зависит от режима Z80 — используйте его для работы с VRAM.
