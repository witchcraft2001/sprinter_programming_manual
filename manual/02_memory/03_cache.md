# 2.3 Кэш-ОЗУ (SRAM) и wait states

> **Навигация:** [← 2.2 DCP маппер](02_dcp_mapper.md) | [Оглавление](../README.md) | [2.4 Flash →](04_flash.md)

---

## Что такое кэш-ОЗУ в Sprinter

На плате Sprinter установлена **микросхема быстрой статической памяти (SRAM)** объёмом **128 КБ**, подключённая к FPGA ACEX. Это **не кэш** в формальном x86-смысле (L1/L2 — прозрачный аппаратный буфер). Это **отдельное быстрое ОЗУ**, в котором Z80 может работать на полной частоте турбо (21 МГц) **без тактов ожидания**.

Кэш-ОЗУ называется «кэшем» по традиции — аналогично кэш-модулям на КР537РУ10 в компьютерах Pentagon-128: та же идея, но реализация другая.

### Ключевые характеристики

| Параметр | Значение |
|---------|---------|
| Тип | SRAM (статическое ОЗУ) |
| Объём | 128 КБ |
| Время доступа | ≤ 15–20 нс (совпадает с CLK42 = 24 нс) |
| Wait states в турбо | **0** (полная скорость 21 МГц) |
| Wait states в DRAM (турбо) | **несколько тактов** (DRAM медленнее) |
| Адресное пространство | WIN0 (`0x0000`–`0x3FFF`) при `CASH_ON=1` |
| Управление | Через `IN A, (#7B)` / `IN A, (#FB)` |

### Сравнение: SRAM vs DRAM vs ROM в турбо (21 МГц)

| Тип памяти | Wait states в турбо | Примечание |
|-----------|--------------------|--------------------|
| **SRAM (кэш)** | **0** | **полные 21 МГц**, максимальная скорость |
| DRAM | несколько CLK42 на цикл обращения | ~10–14 МГц эффективно |
| ROM/Flash | **4 такта CLK42** на каждый доступ | ~95 нс доп. задержки; ускорение от турбо ~1.2–1.5× |

> Значение 4 CLK42 для ROM определено аппаратно (из `SP2_ACEX.TDF`: `WHEN %1XXX => WT_R[] = 4`). При 7 МГц этот же wait укладывается в стандартный Z80-цикл и не ощущается, но при 21 МГц — существенно тормозит.

Для прикладного программиста это означает: **код и данные в SRAM при турбо работают в ~2 раза быстрее**, чем в DRAM, и в ~3 раза быстрее, чем из ROM.

---

## Как устроен кэш аппаратно

FPGA управляет маршрутизацией адресов через сигналы `PRE_ROM`, `PRE_CASH`, `CS_CASH`. Из `SP2_ACEX.TDF`:

```
PRE_ROM  = LCELL(/SYS or A14 or A15 or CASH_ON);
PRE_CASH = LCELL(A14 or A15 or !CASH_ON);

CASE (PRE_ROM, PRE_CASH) IS
    WHEN B"00" => ISA_A[] = B"1000";                         -- ошибка → ISA
    WHEN B"01" => ISA_A[] = ((ROM_RG3 xor !SYS_PG), ROM_RG[2..0]); -- ROM
    WHEN B"10" => ISA_A[] = (B"01", ROM_RG[1..0]);           -- SRAM (кэш)
    WHEN B"11" => ISA_A[] = (!PRE_ISA, GND, ISA_PORT[2..1]); -- ISA
END CASE;

CS_CASHT = DFF((!/RF or !PRE_ROM or PRE_CASH), !/MR,, LCELL(CS_CASHT or !/MR));
CS_CASH  = CS_CASHT & CS_ISA & ISA_A20;
```

Из этого следует:

- **SRAM подключается когда** `CASH_ON=1` **и** адрес в диапазоне `A14=0, A15=0` (то есть `0x0000`–`0x3FFF`, **WIN0**).
- При `CASH_ON=0` тот же диапазон видит ROM или DRAM (в зависимости от SYS-флага).
- Адрес внутри SRAM формируется из `ROM_RG[1..0]` (страница) и адресных линий Z80, что даёт 4 × 16 КБ = **64 КБ** одновременно доступных из 128 КБ.

## Включение и выключение кэша

Из `SP2_ACEX.TDF`:

```
CASH_ON = DFFE(A7, (/IO or /RD), /RESET,, DFF((A[7..0] == B"X1111011"), CLK42,,));
```

Трансляция: при выполнении `IN` из порта с маской `X1111011` (= `#7B` или `#FB`), бит `A7` адреса защёлкивается в `CASH_ON`:

| Операция | Адрес порта | A7 | Результат |
|---------|------------|----|-----------|
| `IN A, (#FB)` | `#FB` = `11111011` | 1 | **CASH_ON = 1** (кэш включён) |
| `IN A, (#7B)` | `#7B` = `01111011` | 0 | **CASH_ON = 0** (кэш выключен) |

```asm
; Включить кэш-ОЗУ
        in      a, (#FB)        ; CASH_ON = 1

; Выключить кэш-ОЗУ (вернуться к DRAM/ROM в WIN0)
        in      a, (#7B)        ; CASH_ON = 0
```

При сбросе (`/RESET`) `CASH_ON` сбрасывается в 0 — кэш выключен.

---

## Влияние на wait states

В `DCP.TDF` сигнал `/CASH` непосредственно входит в формулу memory wait:

```
/MR_WAIT = LCELL(MC_END or MEM_RW or /CASH or (!TURBO & !ACC_ON));
```

- **`/CASH = 0`** (кэш активен для данного адреса) → `/MR_WAIT` **сразу снимается** → **0 wait states**.
- **`/CASH = 1`** (кэш неактивен) → wait определяется MC_END / TURBO / ACC_ON.

В нормальном режиме (7 МГц) терм `(!TURBO & !ACC_ON) = 1` и так снимает wait — разница между SRAM и DRAM несущественна. Но в **турбо (21 МГц)** кэш даёт радикальное ускорение: SRAM отвечает мгновенно, а DRAM ждёт завершения RAS/CAS цикла.

---

## Целевое применение

Кэш-ОЗУ — **важный инструмент для прикладного программиста**, а не только для системного кода. Типичные сценарии:

### Ускорение вычислений

Перенос горячего цикла (декомпрессия, растеризация, математика) в SRAM через WIN0:

```asm
; Включить SRAM-кэш
        in      a, (#FB)

; Скопировать горячий код в WIN0 (0x0000)
        ld      hl, hot_code_src
        ld      de, #0000
        ld      bc, hot_code_len
        ldir

; Вызвать код в SRAM — при турбо 21 МГц выполняется без wait states
        call    #0000

; Выключить кэш
        in      a, (#7B)
```

### Ускорение вывода графики

Подготовить строку/тайл/спрайт в SRAM, затем скопировать в VRAM через акселератор. SRAM → VRAM быстрее, чем DRAM → VRAM, потому что чтение из SRAM не добавляет wait states:

```asm
; Отрисовать спрайт: данные хранятся в SRAM (WIN0),
; VRAM подключена в WIN3
        in      a, (#FB)            ; кэш ON
        ; ... подготовить пиксели в 0x0000..0x00FF ...

        ; Копировать в VRAM через акселератор
        di
        ld      d, d                ; установить размер
        ld      a, sprite_width
        ld      hl, #0000           ; src = SRAM
        ld      de, #C000           ; dst = VRAM (WIN3)
        ld      l, l                ; LD L,L — копировать
        ei
        ld      b, b                ; выключить акселератор
```

### Быстрые таблицы подстановки

Синус-таблицы, таблицы палитры, LUT для эффектов — всё это выигрывает от размещения в SRAM:

```asm
; Скопировать 256-байтную синус-таблицу в SRAM
        in      a, (#FB)
        ld      hl, sin_table_dram
        ld      de, #0100           ; куда в SRAM
        ld      bc, 256
        ldir

; Теперь обращение LD A, (#0100 + X) — без wait states в турбо
```

### Обработка звука в реальном времени

При генерации/обработке PCM-сэмплов для COVOX критичны тайминги. Код генератора в SRAM успевает подготовить сэмплы в темпе DMA:

```asm
; Генератор в SRAM — успевает на 21 МГц
        in      a, (#FB)
        ; ... код синтезатора по адресу #0000 ...
```

## Ограничения

1. **Нельзя вызывать BIOS (`RST #08`) и DSS (`RST #10`) при включённом кэше.** Векторы `RST #08` (адрес `#0008`) и `RST #10` (адрес `#0010`) находятся в WIN0. При `CASH_ON=1` эти адреса указывают в SRAM, а не в ROM/DRAM, где размещён код BIOS и DSS. Любой `RST #08` / `RST #10` при включённом кэше приведёт к исполнению мусора из неинициализированной SRAM. **Перед вызовом BIOS/DSS всегда выключайте кэш:**

    ```asm
    ; ПРАВИЛЬНО: выключить кэш → вызвать DSS → включить обратно
            in      a, (#7B)        ; CASH_ON = 0
            ld      c, #13          ; DSS Read
            rst     #10
            in      a, (#FB)        ; CASH_ON = 1 (если нужен дальше)
    ```

2. **Только WIN0** (`#0000`–`#3FFF`) — кэш-ОЗУ занимает нижние 16 КБ адресного пространства. Код программы (ORG `#8100`, WIN2) и стек (`#BFFF`, WIN2/WIN3) остаются в DRAM.

3. **Одновременно доступно до 64 КБ** из 128 КБ SRAM — через `ROM_RG[1..0]` выбирается одна из 4 страниц по 16 КБ.

4. **При CASH_ON=1 WIN0 «не видит» DRAM** — данные, которые были в DRAM по адресам `#0000`–`#3FFF`, становятся недоступны. Их нужно заранее скопировать, если они нужны.

5. **Прерывания при CASH_ON** — вектор `RST #38` (IM1) и таблица IM2 также находятся в WIN0. Если обработчик прерывания вызывает BIOS/DSS или использует адреса из ROM, **нужно либо выключить кэш перед `EI`**, либо разместить полностью автономный обработчик в SRAM. Типичный приём — `DI` на время работы с кэшем, `EI` после выключения:

    ```asm
            di
            in      a, (#FB)        ; CASH_ON = 1
            ; ... работа в SRAM без прерываний ...
            in      a, (#7B)        ; CASH_ON = 0
            ei
    ```

6. **В нормальном режиме (7 МГц) выигрыш минимален** — DRAM и так успевает за Z80. Кэш актуален именно для **турбо**.

## Wait states — две независимые системы

В Sprinter два независимых механизма вставки тактов ожидания:

1. **I/O wait (`/IO_WAIT`)** — применяется при `IN`/`OUT`. Число тактов из таблицы `W_TAB`, зависит от TYPE-поля DCP и турбо-режима.
2. **Memory wait (`/MR_WAIT`)** — применяется при чтении/записи памяти. Зависит от турбо, акселератора и попадания в кэш-область SRAM.

```
/WAIT = /IO_WAIT & /MR_WAIT
```

## I/O wait states (W_TAB из DCP.TDF)

```
CASE (TURBO, MEM.q[14..12]) IS
    -- TURBO = 0 (7 МГц)               -- TURBO = 1 (21 МГц)
    WHEN 0 => W_TAB[] = 2;  WHEN  8 => W_TAB[] = 2;
    WHEN 1 => W_TAB[] = 2;  WHEN  9 => W_TAB[] = 2;
    WHEN 2 => W_TAB[] = 1;  WHEN 10 => W_TAB[] = 4;
    WHEN 3 => W_TAB[] = 1;  WHEN 11 => W_TAB[] = 4;
    WHEN 4 => W_TAB[] = 1;  WHEN 12 => W_TAB[] = 7;
    WHEN 5 => W_TAB[] = 2;  WHEN 13 => W_TAB[] = 7;
    WHEN 6 => W_TAB[] = 7;  WHEN 14 => W_TAB[] = 7;
    WHEN 7 => W_TAB[] = 10; WHEN 15 => W_TAB[] = 10;
END CASE;
```

### Нормальный режим (7 МГц)

| MEM.q[14..12] | Waits | Примеры портов (DCP.MIF) |
|---------------|-------|--------------------------|
| 0 | 2 | — (не назначен) |
| 1 | 2 | `1000` — незанятые порты (default) |
| 2 | 1 | `2000` — запись: банки RAM, AY, COVOX, ROM `#8F` |
| 3 | 1 | `3000` — чтение AY (`#52`, `#8C`) |
| 4 | 1 | `4000` — чтение клавиатуры, `C000` — системные порты |
| 5 | 2 | `5000` — Kempston Mouse (`#58`) |
| 6 | 7 | `6000` — HDD CS / ZX-видео |
| 7 | 10 | `7000` — MAX7000 (FDD, CMOS, HDD switches) |

### Турбо-режим (21 МГц)

| MEM.q[14..12] | Waits | Комментарий |
|---------------|-------|-------------|
| 0 | 2 | — |
| 1 | 2 | default |
| 2 | **4** | банки, AY/COVOX запись |
| 3 | **4** | AY read |
| 4 | **7** | клавиатура, системные порты |
| 5 | **7** | Kempston Mouse |
| 6 | 7 | HDD CS / ZX-видео |
| 7 | 10 | MAX7000 |

### HDD данные

```
CASE LCELL(MEM.q[11..8] == 0) IS
    WHEN 0 => HDD_W[] = 10;   -- HDD регистры
    WHEN 1 => HDD_W[] =  4;   -- HDD данные (1F0h)
END CASE;
```

> **Важно:** W_TAB — это **I/O waits** (`IN`/`OUT`). Обращения к памяти проходят через `/MR_WAIT`.

---

## Memory wait states (/MR_WAIT)

```
/MR_WAIT = LCELL(MC_END or MEM_RW or /CASH or (!TURBO & !ACC_ON));
```

| Ситуация | Memory wait |
|---------|-------------|
| 7 МГц, любая память | **≈ 0** (DRAM и Flash успевают за Z80) |
| 21 МГц, адрес в **SRAM** (`CASH_ON=1`, WIN0) | **0** — полные 21 МГц |
| 21 МГц, адрес в **DRAM** | Wait до `MC_END` (несколько CLK42 тактов) |
| 21 МГц, ROM/Flash | **4 CLK42** (~95 нс) на каждый доступ (`WT_R[] = 4` в SP2_ACEX.TDF) |
| 21 МГц, акселератор (`ACC_ON=1`) | Акселератор управляет тактами |

## Типичные тайминги инструкций

| Инструкция | T-states | 7 МГц, мкс | 21 МГц SRAM, мкс | 21 МГц DRAM, мкс |
|-----------|----------|-----------|------------------|------------------|
| `NOP` | 4 | 0.57 | **0.19** | 0.19 |
| `LD r, (HL)` | 7 | 1.00 | **0.33** | ~0.5–0.7 |
| `JP nn` | 10 | 1.43 | **0.48** | ~0.7–1.0 |
| `CALL nn` | 17 | 2.43 | **0.81** | ~1.2–1.7 |
| `LDIR` (байт) | 21 | 3.00 | **1.00** | ~1.5–2.0 |

В SRAM при турбо — полное 3-кратное ускорение. В DRAM при турбо — реально ~1.5–2×.

---

## Видеоконфликты в ZX-режиме

В ZX-совместимом режиме (`SC0=1`) CPU и видеоконтроллер конкурируют за доступ к странице 5 (ZX-экран, `0x4000`–`0x7FFF`) — аппаратура вставляет wait до конца видео-транзакции (contended memory). В нативном режиме (`SC0=0`) конфликтов нет: VRAM и DRAM — разные чипы.

## Пример: полный паттерн работы с кэшем

```asm
; Ускорить распаковку данных через SRAM-кэш

DecompressInCache:
        ; 1. Включить кэш
        in      a, (#FB)            ; CASH_ON = 1

        ; 2. Скопировать распаковщик в SRAM (WIN0)
        ld      hl, Decompressor    ; код в DRAM
        ld      de, #0000           ; целевой адрес в SRAM
        ld      bc, DecompressorLen
        ldir

        ; 3. Скопировать упакованные данные в SRAM
        ld      hl, packed_data
        ld      de, #1000           ; после кода распаковщика
        ld      bc, packed_len
        ldir

        ; 4. Вызвать распаковщик — работает на полных 21 МГц
        call    #0000

        ; 5. Скопировать результат обратно в DRAM (или в VRAM)
        ; ... (здесь кэш всё ещё включён, WIN0 = SRAM) ...

        ; 6. Выключить кэш
        in      a, (#7B)            ; CASH_ON = 0
        ; Теперь WIN0 снова видит DRAM
        ret

Decompressor:
        ; ... код распаковщика ...
        ret
DecompressorLen equ $ - Decompressor
```

## Ключевые моменты

> - Кэш-ОЗУ в Sprinter — это **SRAM 128 КБ**, отдельный чип на плате (не DRAM, не внутри FPGA).
> - Обеспечивает **0 wait states при 21 МГц** — полное 3× ускорение турбо.
> - Подключается в WIN0 (`#0000`–`#3FFF`) при `CASH_ON=1`.
> - Включение: `IN A, (#FB)`; выключение: `IN A, (#7B)`.
> - **При включённом кэше нельзя вызывать BIOS/DSS** (`RST #08`, `RST #10`) — их векторы в WIN0 указывают в SRAM, а не в ROM. Выключайте кэш перед вызовом.
> - ROM/Flash при турбо добавляет **4 CLK42** (~95 нс) на каждый доступ — ускорение от турбо лишь ~1.2–1.5×.
> - **Актуально для прикладных программ:** размещайте в SRAM горячие циклы, LUT-таблицы, спрайтовые буферы для максимальной скорости.
> - В нормальном режиме (7 МГц) разница минимальна — кэш критичен именно для турбо.
