## Проблема: клавиатура влияет на тайминг кадровых прерываний

### Суть проблемы

Сигнал `(SVIDEO.INTT & KEYS.int)` используется как **тактовый вход** (CLK) триггера, формирующего INT (см. [9.1 формула INT_X](01_interrupts.md#формирование-сигнала-int--аппаратная-схема)). Прерывание CPU происходит по **фронту**, когда **оба** сигнала одновременно HIGH. Если в момент VSync клавиатурный сигнал `KEYS.int` находится в LOW (идёт приём очередного бита PS/2), фронт INT сдвигается до момента, когда `KEYS.int` снова станет HIGH.

Максимальный джиттер = **один период клавиатурного цикла** ≈ 460 мкс (почти полмиллисекунды). При активном наборе текста или удержании клавиши PS/2-контроллер непрерывно шлёт скан-коды, и фронт INT «плавает» от кадра к кадру.

### Симптомы

- **Неравномерная анимация** — кадры отрисовываются с разной длительностью (19.5–20.5 мс вместо стабильных 20 мс).
- **Дрожание звука** — если AY-плеер вызывается из обработчика INT, тайминг нот плывёт.
- **Неточные таймеры** — подсчёт кадров как меры времени даёт ошибку.

### Решение 1: отключить клавиатурное прерывание через ALL_MODE

Бит 3 регистра `ALL_MODE` (`#C3`) управляет `KEYS.int_ena`. Сбросив его, вы отсоединяете клавиатуру от формулы INT:

```asm
; Отключить влияние клавиатуры на INT
        in      a, (#C3)
        and     %11110111           ; сбросить бит 3
        out     (#C3), a

; Теперь INT приходит строго по VSync ~50 Гц
; Клавиатуру читать через SIO (порт #18/#19) вручную
```

> PS/2 клавиатура **продолжает работать** — данные приходят через SIO Z84C15 (порты `#18`/`#19`). Отключается только влияние клавиатурного сигнала на тактирование INT.

### Решение 2: CTC как стабильный таймер

CTC Z84C15 не проходит через формулу `INT_X` в FPGA — его прерывания идут напрямую через Z84C15:

```asm
CTC_CH2         EQU #12

; Настроить CTC канал 2 на ~50 Гц
        ld      a, #57              ; Timer, prescaler 256, INT enable
        out     (CTC_CH2), a
        ld      a, 112              ; константа (подбирается)
        out     (CTC_CH2), a
```

### Решение 3: vsync_flag — подпрограмма ожидания кадрового прерывания

SDK настраивает обработчик VSync, который **выставляет флаг**. Вызывающий код ждёт этот флаг, а не просто `HALT`:

```asm
; === Обработчик VSync ===
im2VShandler:
        push    af
        ; ... обновление палитры, звука ...

        ld      a, 1
        ld      (vsync_flag), a     ; <-- ВЫСТАВЛЯЕМ ФЛАГ

        pop     af
        ei
        reti

vsync_flag:     db 0
```

```asm
; === Подпрограмма ожидания кадрового прерывания ===
waitVsync:
        xor     a
        ld      (vsync_flag), a     ; сбросить флаг
.loop:
        ei
        halt                        ; CPU спит до любого прерывания

        ld      a, (vsync_flag)     ; проверить: это было VSync?
        and     a
        jr      z, .loop            ; нет (клавиатурное/CTC) → ждать дальше

        ret                         ; да → VSync произошёл
```

**Зачем цикл, а не просто `HALT`?** `HALT` пробуждается **любым** прерыванием. Без проверки `vsync_flag` программа может начать рисовать кадр раньше обратного хода луча → tearing. Цикл гарантирует продолжение **только после настоящего VSync**.

## Синхронизация с обратным ходом луча

### Tearing и glitch

Видеоконтроллер непрерывно читает VRAM строка за строкой. Если программа пишет в VRAM **в момент, когда луч проходит через обновляемую область**, на экране появляется **tearing** (горизонтальный разрыв: часть кадра старая, часть новая) или **glitch** (мусор на несколько пикселей).

### Безопасное окно: VBI

Между последней видимой строкой и первой строкой следующего кадра — период **обратного хода луча** (vertical blanking interval, VBI): около **1.5–2 мс** при 50 Гц PAL. Кадровое прерывание (VSync) приходит в начале VBI.

> **Правило:** обновляйте VRAM (спрайты, фон, палитру, PORT_Y) **сразу после VSync-прерывания**.

### Паттерн: double buffering + flip по VSync

Самый надёжный способ — полностью исключить tearing:

```asm
GameLoop:
        ; 1. Рисовать в скрытый буфер (сколько угодно времени)
        call    ClearHiddenBuffer
        call    DrawSprites
        call    DrawUI

        ; 2. Дождаться VSync
        call    waitVsync

        ; 3. Flip (в обработчике VSync или здесь)
        in      a, (#C9)
        xor     1
        out     (#C9), a

        jr      GameLoop
```

Пользователь видит только полностью готовые кадры.

### Без double buffering

Единственный вариант — **успеть обновить VRAM за VBI** (~1.5 мс = ~10 500 тактов при 7 МГц, ~31 500 при 21 МГц). Для текстового режима (несколько символов) этого хватает. Для полной перерисовки — обновляйте частями, по нескольким строкам за кадр.

### CTC и VSync: разделение ответственности

CTC работает от кварца и **не синхронизирован с лучом**. CTC-прерывание может попасть в середину видимой части кадра — запись в VRAM в этот момент вызовет glitch.

**Правило: CTC для логики, VSync для графики.**

```asm
; CTC-обработчик: логика, звук, ввод — БЕЗ записи в VRAM
CTC_Handler:
        call    ReadKeyboard
        call    UpdateGameLogic
        call    PlayMusic
        ei
        reti

; VSync-обработчик: flip, палитра — ЗДЕСЬ пишем в VRAM
VSync_Handler:
        in      a, (#C9)
        xor     1
        out     (#C9), a
        call    UpdatePalette
        ld      a, 1
        ld      (vsync_flag), a
        ei
        reti
```

## Настройка VSync-обработчика в SDK (lib_startup.asm)

SDK из `zx-sprinter-sdk` использует продуманную схему: **два раздельных обработчика** с детерминированием источника и синхронизацией CTC с VSync.

### Шаг 1: размещение таблицы и точки входа

```asm
; lib_startup.asm, строки 827–831:
        org     #FDFD
        jp      im2VShandler        ; точка входа по умолчанию

        org     #FE00
im2vector:
        ds      257, #FD            ; 257 байт, все = #FD
```

### Шаг 2: синхронизация CTC с фронтом VSync

При старте SDK определяет момент VSync и запускает CTC именно в нём:

```asm
; lib_startup.asm, строки 116–136:

        ; Временно: пустой обработчик на #FDFE, VSync на #FE06
        ld      hl, im2empty : ld (#FDFE), hl
        ld      hl, im2VShandler : ld (#FE06), hl

        ; Цикл: ждём именно VSync (не клавиатуру)
.sync:
        ei : halt : di
        in      a, (SIO_CONTROL_A)
        bit     0, a                ; SIO бит 0 = байт пришёл?
        jr      z, .endsync         ; нет → это VSync
        call    PS2Scan             ; да → клавиатура, обработать и ждать
        jr      .sync

.endsync:
        ; Мы на фронте VSync — запускаем CTC
        ld      a, #57 : out (CTC_CH2), a
        ld      a, 112 : out (CTC_CH2), a
        ld      a, #D7 : out (CTC_CH3), a
        ld      a, 160 : out (CTC_CH3), a
        ld      a, 0   : out (CTC_CH0), a

        ; Переключаем вектор на клавиатурный обработчик
        ld      hl, im2Keyhandler : ld (#FDFE), hl
```

**Что происходит:** SDK крутит `HALT`, после каждого пробуждения проверяет SIO. Если там байт — это клавиатура; если нет — VSync. **Именно на фронте VSync** запускается CTC → он синхронизирован с началом кадра.

### Шаг 3: два вектора — два обработчика

| Адрес | Вектор от | Обработчик | Задача |
|-------|----------|-----------|--------|
| `#FDFE` | CTC/клавиатура | `im2Keyhandler` | PS/2 сканирование |
| `#FE06` | VSync (FPGA) | `im2VShandler` | Звук, палитра, flip, `vsync_flag` |

Разные устройства кладут разные значения вектора на шину → разные адреса в таблице → разные обработчики.

### Шаг 4: полный VSync-обработчик

Из `lib_startup.asm` (строки 467–656):

```asm
im2VShandler:
        push    af : push bc : push de : push hl : push ix : push iy
        exa : exx
        push    af : push bc : push de : push hl

        ; 1. PORT_Y в безопасную зону
        in      a, (PORT_Y) : push af
        ld      a, #C0 : out (PORT_Y), a

        ; 2. Обновить AY (звук)
        ld      a, (SNDFLAG)
        ; ... выбор чипа, вывод AYREGS ...

        ; 3. Сохранить страницы, переключить на звуковые данные
        in      a, (PAGE1) : ld (_memSlot1), a
        ld      a, (memSNDPage) : out (PAGE1), a

        ; 4. Вызвать музыкальный плеер
        call    #400C               ; обновить трек
        call    #4006               ; микшировать

        ; 5. Восстановить страницы
        ld      a, (_memSlot1) : out (PAGE1), a

        ; 6. Счётчик кадров (32-бит)
        ld      hl, _time
        ld      b, 4
.time1: inc     (hl)
        jr      nz, .time2
        inc     hl
        djnz    .time1
.time2:

        ; 7. Восстановить PORT_Y
        pop     af
        out     (PORT_Y), a

        ; 8. Выставить флаг
        ld      a, #FF : ld (vsync_flag), a

        ; 9. Обработать клавиатуру
        call    PS2Scan

        ; 10. Восстановить регистры
        pop     hl : pop de : pop bc : pop af
        exx : exa
        pop     iy : pop ix : pop hl : pop de : pop bc : pop af
        ei
        reti
```

### Важные детали

- **PORT_Y сохраняется/восстанавливается** — обработчик уводит PORT_Y в `#C0`, потом возвращает.
- **Страницы окон сохраняются** — PAGE1/PAGE2 переключаются для доступа к звуковым данным.
- **`vsync_flag = #FF`** — подпрограмма `waitVsync` ждёт именно этот байт.
- **32-битный `_time`** — ~20 мс на тик (50 Гц); `_time / 50 = секунды`.

## Ключевые моменты

> - Клавиатура вносит джиттер до 460 мкс в кадровое прерывание. Отключается через бит 3 порта `#C3`.
> - **waitVsync:** цикл `HALT` + `vsync_flag` — единственный надёжный способ дождаться VSync.
> - **Tearing:** обновляйте VRAM только в VBI (после VSync) или через double buffering.
> - **CTC для логики, VSync для графики** — CTC не синхронизирован с лучом.
> - SDK синхронизирует CTC с VSync при старте (цикл определения источника через SIO).
> - VSync-обработчик SDK: PORT_Y в безопасную зону → звук → flip → vsync_flag → PS2Scan.
