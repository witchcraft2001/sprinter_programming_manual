## Режимы прерываний Z80

Z80 поддерживает 3 режима прерываний:

| Режим | Инструкция | Поведение |
|-------|-----------|-----------|
| IM 0 | `IM 0` | Устройство кладёт опкод на шину, CPU исполняет |
| IM 1 | `IM 1` | При прерывании → `RST #38` |
| IM 2 | `IM 2` | Векторный режим: I × 256 + vector |

В Sprinter чаще всего используется **IM 2** — он позволяет разделить прерывания от разных источников (VSync, CTC, SIO).

---

## Источники прерываний

В Sprinter существует несколько аппаратных источников прерываний:

| Источник | Частота | Сигнал в FPGA | Описание |
|----------|---------|--------------|----------|
| **Кадровое (VSync)** | ~50 Гц (PAL) | `SVIDEO.INTT` | Конец кадра — основной таймер |
| **Клавиатурное (PS/2)** | ~2.1 кГц | `KEYS.int` | Счётчик бит PS/2 достиг 0 |
| **CTC** (4 канала) | настраивается | через Z84C15 | Пользовательские таймеры |
| **SIO Rx** | по событию | через Z84C15 | Получен байт от PS/2 |
| **CBL (COVOX)** | настраивается | `CBL_INT` | Буфер аудио опустошён наполовину |
| **NMI** | по событию | `NMI_X` | Alt+F12 (если включено в прошивке) |
| **ISA** | по событию | через ISA-слот | IRQ от ISA-устройства |

## Формирование сигнала INT — аппаратная схема

Сигнал `/INT` процессора Z80 формируется в FPGA (`SP2_ACEX.TDF`) из **трёх** независимых источников:

```
INT_X = !DFF(GND, (SVIDEO.INTT & KEYS.int), ...) or !CBL_INT;
```

Разберём подробно:

### 1. Кадровое прерывание (SVIDEO.INTT)

Генерируется в видеоконтроллере (`VIDEO2.TDF`) при достижении определённой позиции растра:

```
INTTX = DFF((BORD & MODE03 & MODE02 & MODE00), LWR_COL,,);
INTT  = DFF(!(INTTX & (CTV[2..0] == 7)), CT5,,);
```

Срабатывает **один раз за кадр** (~50 Гц PAL). Привязано к позиции бордюра — аналогично ZX Spectrum.

### 2. Клавиатурное прерывание (KEYS.int)

Генерируется в модуле клавиатуры (`KBD.TDF`):

```
INT = DFF((KB_CT[] == 0), CLK42,, INT_ENA);
```

`KB_CT` — счётчик бит PS/2 (7→0), тактируемый CLK_K (~15 кГц). Прерывание возникает каждые **~460 мкс** (7 тактов × 67 мкс). Управляется через `INT_ENA`:

```
; SP2_ACEX.TDF, строка 325:
KEYS.int_ena = LCELL(ALL_MODE0 & ALL_MODE3);
```

Клавиатурное прерывание **включено только** при `ALL_MODE0 = 1` **и** `ALL_MODE3 = 1` (регистр `ALL_MODE`, порт `#C3`).

### 3. COVOX/CBL прерывание (CBL_INT)

Генерируется при опустошении половины FIFO-буфера COVOX:

```
CBL_INT = DFF(GND, !CBL_CNT6,, (CBL_INT_ENA & (/IO or /M1)));
```

Включается битом 4 порта `#89` (`CBL_INT_ENA`).

### Итоговая формула

Все три источника объединяются в один сигнал `/INT`:

```
INT_X = !DFF(GND, (SVIDEO.INTT & KEYS.int), acknowledge) or !CBL_INT;
```

Обратите внимание: кадровое и клавиатурное прерывания соединены через **AND** — это ключевая особенность, порождающая проблему нестабильных прерываний.

## Проблема: клавиатура влияет на тайминг кадровых прерываний

### Суть проблемы

Сигнал `(SVIDEO.INTT & KEYS.int)` используется как **тактовый вход** (CLK) триггера, формирующего INT. Это означает: прерывание CPU происходит по **фронту**, когда **оба** сигнала одновременно HIGH. Если в момент VSync клавиатурный сигнал `KEYS.int` находится в LOW (идёт приём очередного бита PS/2), фронт INT сдвигается до момента, когда `KEYS.int` снова станет HIGH.

Максимальный джиттер = **один период клавиатурного цикла** ≈ 460 мкс (почти полмиллисекунды). При активном наборе текста или удержании клавиши PS/2-контроллер непрерывно шлёт скан-коды, и фронт INT «плавает» от кадра к кадру.

### Симптомы

- **Неравномерная анимация** — кадры отрисовываются с разной длительностью (19.5–20.5 мс вместо стабильных 20 мс).
- **Дрожание звука** — если AY-плеер вызывается из обработчика INT, тайминг нот плывёт.
- **Неточные таймеры** — подсчёт кадров как меры времени даёт ошибку.

### Решения

#### 1. Отключить клавиатурное прерывание через ALL_MODE

Бит 3 регистра `ALL_MODE` (`#C3`) управляет `KEYS.int_ena`. Сбросив его, вы отсоединяете клавиатуру от формулы INT, и кадровое прерывание становится стабильным:

```asm
; Отключить клавиатурные прерывания для стабильного VSync
        in      a, (#C3)
        and     %11110111           ; сбросить бит 3
        out     (#C3), a

; Теперь INT приходит строго по VSync ~50 Гц
; Клавиатуру читать через SIO (порт #18/#19) вручную в обработчике
```

> **Важно:** при отключённом `ALL_MODE3` PS/2 клавиатура **продолжает работать** — данные приходят через SIO Z84C15 (порты `#18`/`#19`). Отключается только влияние клавиатурного сигнала на тактирование INT.

#### 2. Использовать CTC как стабильный источник

CTC Z84C15 генерирует прерывания от кварцевого генератора, независимо от клавиатуры:

```asm
CTC_CH2         EQU #12

; Настроить CTC канал 2 на ~50 Гц
; 7 000 000 / (256 × 547) ≈ 50 Гц
        ld      a, #57              ; Timer, prescaler 256, INT enable
        out     (CTC_CH2), a
        ld      a, 547/256          ; аппроксимация (подбирается под прошивку)
        out     (CTC_CH2), a
```

CTC прерывание не зависит от `KEYS.int` — оно приходит строго по таймеру.

#### 3. Паттерн из SDK: vsync_flag

SDK (`lib_startup.asm`) настраивает обработчик VSync, который **выставляет флаг**. Главный цикл ждёт этот флаг, а не просто `HALT`:

```asm
; === Обработчик VSync (вызывается из IM2 при кадровом прерывании) ===
im2VShandler:
        push    af
        ; ... обновление палитры, звука, спрайтов ...

        ld      a, 1
        ld      (vsync_flag), a     ; <-- ВЫСТАВЛЯЕМ ФЛАГ

        pop     af
        ei
        reti

vsync_flag:     db 0
```

```asm
; === Подпрограмма ожидания кадрового прерывания ===
waitVsync:
        xor     a
        ld      (vsync_flag), a     ; сбросить флаг
.loop:
        ei
        halt                        ; CPU спит до любого прерывания

        ld      a, (vsync_flag)     ; проверить: это было VSync?
        and     a
        jr      z, .loop            ; нет (клавиатурное или CTC) → ждать дальше

        ret                         ; да → VSync произошёл, продолжить
```

**Зачем цикл, а не просто `HALT`?** Потому что `HALT` пробуждается **любым** прерыванием — клавиатурным, CTC, COVOX. Если после `HALT` не проверить `vsync_flag`, программа может начать рисовать новый кадр раньше, чем завершился обратный ход луча → артефакты (tearing). Цикл с `vsync_flag` гарантирует, что продолжение произойдёт **только после настоящего VSync**.

## CTC — независимый таймер Z84C15

CTC не проходит через формулу `INT_X` в FPGA — его прерывания идут напрямую через шину Z84C15. Это самый стабильный источник прерываний.

### Инициализация из SDK

```asm
CTC_CH0     EQU #10
CTC_CH2     EQU #12
CTC_CH3     EQU #13

; SDK lib_startup.asm, строки 130-134:
        ld      a, #57 : out (CTC_CH2), a    ; Timer, prescaler 256, INT enable
        ld      a, 112 : out (CTC_CH2), a    ; константа счёта

        ld      a, #D7 : out (CTC_CH3), a    ; Timer, prescaler 256, INT enable
        ld      a, 160 : out (CTC_CH3), a    ; константа счёта
```

### CTC вектор прерывания

В IM2 CTC автоматически формирует вектор. Базовый вектор задаётся первой записью с бит 0 = 0:

```asm
        ld      a, #80              ; базовый вектор CTC
        out     (CTC_CH0), a
; 4 канала → векторы: #80, #82, #84, #86
```

## Синхронизация с обратным ходом луча (VSync)

### Проблема: tearing и glitch

Видеоконтроллер Sprinter непрерывно читает VRAM и формирует видеосигнал строка за строкой. Если программа записывает данные в VRAM **в момент, когда луч проходит через обновляемую область**, на экране появляется **tearing** (горизонтальный разрыв) или **glitch** (кратковременный мусор) — часть кадра показывает старые данные, часть — новые.

### Безопасное окно: обратный ход луча

Между последней видимой строкой кадра и первой строкой следующего кадра есть период **обратного хода луча** (vertical blanking interval, VBI) — около **1.5–2 мс** при 50 Гц PAL. В это время видеоконтроллер не читает VRAM для отображения, и можно безопасно обновлять данные.

Кадровое прерывание (VSync) приходит именно в начале этого периода. Правило:

> **Обновляйте VRAM (спрайты, фон, палитру) сразу после VSync-прерывания** — это единственный гарантированно безопасный момент.

### Паттерн: обновление в обработчике VSync

```asm
; Обработчик VSync — вызывается в начале обратного хода
im2VShandler:
        push    af
        push    bc
        push    de
        push    hl

        ; 1. Flip double buffer (мгновенно, без tearing)
        in      a, (#C9)
        xor     1
        out     (#C9), a

        ; 2. Обновить палитру (если изменилась)
        call    UpdatePalette

        ; 3. Обновить AY регистры (звук)
        call    PlayMusic

        ; 4. Выставить флаг для главного цикла
        ld      a, 1
        ld      (vsync_flag), a

        pop     hl
        pop     de
        pop     bc
        pop     af
        ei
        reti
```

### Паттерн: рисование в скрытый буфер, flip по VSync

С double buffering программа рисует **вне обработчика** (в главном цикле), но flip делает **строго по VSync**:

```asm
GameLoop:
        ; 1. Рисовать в скрытый буфер (сколько угодно времени)
        call    ClearHiddenBuffer
        call    DrawSprites
        call    DrawUI

        ; 2. Дождаться VSync
        call    waitVsync           ; блокирует до настоящего VSync

        ; 3. Flip уже произошёл в im2VShandler (см. выше)
        ;    Теперь то, что мы рисовали, стало видимым
        ;    А мы начинаем рисовать в другой буфер

        jr      GameLoop
```

Это полностью исключает tearing: пользователь видит только полностью готовые кадры.

### Если нет double buffering

Без двойной буферизации (например, в текстовом режиме) единственный вариант — **успеть обновить VRAM за время VBI** (~1.5 мс). При 7 МГц это ~10 500 тактов, при 21 МГц — ~31 500. Для текстового режима (запись нескольких символов) этого обычно достаточно. Для полной перерисовки экрана — нет; тогда обновляйте постепенно, по нескольким строкам за кадр.

### CTC и VSync: синхронизация

CTC работает от кварца и **не синхронизирован с обратным ходом луча**. Если использовать CTC как единственный таймер (без `vsync_flag`), прерывания будут приходить стабильно по частоте, но в **произвольной фазе** относительно VSync. Это означает: CTC-прерывание может попасть в середину видимой части кадра, и обновление VRAM в этот момент вызовет glitch.

**Правильный подход: CTC для логики, VSync для отрисовки.**

```asm
; CTC-обработчик: логика, звук, клавиатура — БЕЗ записи в VRAM
CTC_Handler:
        call    ReadKeyboard
        call    UpdateGameLogic
        call    PlayMusic
        ei
        reti

; VSync-обработчик: flip, палитра, спрайты — ЗДЕСЬ пишем в VRAM
VSync_Handler:
        ; flip double buffer
        in      a, (#C9)
        xor     1
        out     (#C9), a
        ; обновить палитру
        call    UpdatePalette
        ; выставить флаг
        ld      a, 1
        ld      (vsync_flag), a
        ei
        reti
```

Если используются два раздельных вектора IM2 (CTC и VSync), каждый обработчик делает свою работу:
- **CTC** (~50 Гц, стабильный): логика, звук, ввод — не трогает VRAM
- **VSync** (~50 Гц, привязан к лучу): flip, палитра, PORT_Y — работа с VRAM

## CBL (COVOX) прерывание

Включается битом 4 порта `#89`:

```asm
        ld      a, #9C              ; COVOX ON + INT enable + 31.25 кГц
        out     (#89), a
```

Прерывание приходит при опустошении половины 256-байтного FIFO. В обработчике программа дозаписывает следующую порцию сэмплов. Подробнее — в разделе [6.3 COVOX DMA](../06_audio/03_covox_dma.md).

## NMI

Non-Maskable Interrupt в Sprinter:

```
; SP2_ACEX.TDF:
IF (NMI_ON == "ON") GENERATE
    NMI_X = (!KEYS.kb_f12 & KEYS.kb_alt);   ; Alt+F12
END GENERATE;
```

По умолчанию NMI **отключён** в прошивке (`NMI_ON = "OFF"`). Если включён — срабатывает по Alt+F12, переход на `RST #66`.

## IM 1 — простой режим

При IM 1 все прерывания идут на `RST #38` (адрес `0x0038`).

```asm
        im      1
        ei

; Обработчик по адресу 0x0038
        org     #0038
        push    af
        ; ... обработка ...
        pop     af
        ei
        reti
```

Проблема IM 1: невозможно различить источники — одна точка входа для всех прерываний.

## IM 2 — векторный режим

В IM 2 Z80 вычисляет адрес обработчика по формуле:
```
Vector_addr = (I << 8) | Vector_byte
```

где `I` — регистр I (старший байт), `Vector_byte` — значение, которое устройство кладёт на шину данных при подтверждении прерывания.

По этому адресу лежит **16-битный указатель** на обработчик. Z80 читает его и переходит туда.

### Проблема в Sprinter

В IM 2 при подтверждении прерывания устройство кладёт на шину данных **вектор-байт**. Z80 читает 16-битный указатель из адреса `I × 256 + вектор`. Проблема: разные устройства (CTC канал 0, канал 1, SIO, FPGA) кладут **разные** значения вектора. Если выделять отдельный обработчик на каждый вектор — нужна сложная диспетчеризация.

**Трюк с таблицей 257 одинаковых байт** решает проблему: таблица заполняется одним и тем же значением (например, `#FD`). Тогда **любой** вектор-байт, считанный с шины, даст один и тот же адрес обработчика.

Почему 257 байт, а не 256? Потому что вектор-байт может быть нечётным — тогда Z80 читает слово, начиная с нечётного адреса. Если на границе таблицы (позиция 255) будет `#FD`, а байт 256 — мусор, получится неверный адрес. Лишний 257-й байт гарантирует, что чтение с позиции 255 даст `#FDFD`, а не `#FD??`.

Пример: `I = #FE`, таблица на `#FE00`–`#FF00` заполнена `#FD`. Любой вектор `V` → адрес `#FE00 + V` → читается слово `#FDFD` → переход на обработчик по адресу, записанному в `#FDFD`.

## Стандартная настройка IM 2 в Sprinter

### Таблица векторов на #FE00

```asm
        ; 1. Заполнить 257 байт значением #FD
        ld      hl, #FE00
        ld      de, #FE01
        ld      bc, 256             ; 256 + 1 = 257 байт
        ld      (hl), #FD
        ldir

        ; 2. По адресу #FDFD записать указатель на наш обработчик
        ;    (Z80 прочитает слово #FDFD → int_handler)
        ld      hl, int_handler
        ld      (#FDFD), hl

        ; 3. Установить I = #FE → таблица начинается с #FE00
        ld      a, #FE
        ld      i, a

        im      2
        ei
```

Как это работает:

```
Любое устройство кладёт вектор V (0..255) на шину данных
    ↓
Z80 вычисляет: адрес = I × 256 + V = #FE00 + V
    ↓
По адресу #FE00+V лежит байт #FD
По адресу #FE00+V+1 тоже #FD
    ↓
Z80 читает 16-битное слово: #FDFD
    ↓
Z80 переходит на адрес, записанный в (#FDFD): JP int_handler
```

### Обработчик

```asm
int_handler:
        push    af
        push    bc
        push    de
        push    hl
        push    ix
        push    iy
        ex      af, af'
        push    af
        exx
        push    bc
        push    de
        push    hl

        ; ... основная обработка ...
        call    VSyncHandler

        pop     hl
        pop     de
        pop     bc
        exx
        pop     af
        ex      af, af'
        pop     iy
        pop     ix
        pop     hl
        pop     de
        pop     bc
        pop     af
        ei
        reti
```

## Настройка VSync-обработчика в SDK (lib_startup.asm)

SDK из `zx-sprinter-sdk` использует продуманную схему: **два раздельных обработчика** с детерминированием источника прерывания и синхронизацией CTC с VSync.

### Шаг 1: размещение таблицы векторов и точки входа

```asm
; lib_startup.asm, строки 827–831:
        org     #FDFD
        jp      im2VShandler        ; точка входа по умолчанию

        org     #FE00
im2vector:
        ds      257, #FD            ; 257 байт, все = #FD
```

Таблица `#FE00`–`#FF00` заполнена `#FD`. Любой вектор → чтение слова по `#FE00 + V` → `#FDFD` → `JP im2VShandler`. По умолчанию **все** прерывания попадают в `im2VShandler`.

### Шаг 2: определение источника и синхронизация CTC

При старте SDK выполняет процедуру синхронизации CTC с VSync:

```asm
; lib_startup.asm, строки 116–136:

        ; Сначала в #FDFE ставим пустой обработчик
        ld      hl, im2empty : ld (#FDFE), hl
        ; В #FE06 ставим VSync-обработчик
        ld      hl, im2VShandler : ld (#FE06), hl

        ; Цикл синхронизации: ждём именно VSync (не клавиатуру)
.sync:
        ei : halt : di
        in      a, (SIO_CONTROL_A)
        bit     0, a                ; SIO бит 0 = байт пришёл?
        jr      z, .endsync         ; нет → это было VSync-прерывание
.keys:
        call    PS2Scan             ; да → это клавиатура, обработать и ждать дальше
        jr      .sync

.endsync:
        ; Теперь мы точно на фронте VSync — настраиваем CTC
        ld      a, #57 : out (CTC_CH2), a    ; Timer, prescaler 256, INT enable
        ld      a, 112 : out (CTC_CH2), a
        ld      a, #D7 : out (CTC_CH3), a
        ld      a, 160 : out (CTC_CH3), a
        ld      a, 0   : out (CTC_CH0), a

        ; Теперь переключаем вектор #FDFE на клавиатурный обработчик
        ld      hl, im2Keyhandler : ld (#FDFE), hl
```

**Что здесь происходит:**

1. SDK входит в цикл `HALT` и после каждого пробуждения проверяет, **от кого** пришло прерывание: если SIO содержит байт (бит 0 = 1) — это клавиатура, обрабатываем и ждём дальше.
2. Когда `SIO бит 0 = 0` — значит пробуждение было **не от клавиатуры**, а от VSync.
3. **Именно в этот момент** (на фронте VSync) SDK запускает CTC — так CTC оказывается **синхронизирован с началом кадра**.
4. После синхронизации вектор `#FDFE` переключается с `im2empty` на `im2Keyhandler`.

### Шаг 3: два вектора — два обработчика

SDK использует **разные адреса** в таблице `#FE00` для разных источников:

| Адрес в таблице | Вектор | Обработчик | Назначение |
|----------------|--------|-----------|-----------|
| `#FDFE` | CTC/клавиатура | `im2Keyhandler` | PS/2 сканирование |
| `#FE06` | VSync | `im2VShandler` | Звук, палитра, flip, счётчик кадров |

Z84C15 CTC кладёт на шину свой вектор (определяемый базовым вектором CTC), а FPGA — свой. Разные значения → разные адреса в таблице → разные обработчики.

### Шаг 4: VSync-обработчик (im2VShandler)

Полная структура из `lib_startup.asm` (строки 467–656):

```asm
im2VShandler:
        push    af : push bc : push de : push hl : push ix : push iy
        exa : exx
        push    af : push bc : push de : push hl

        ; 1. Увести PORT_Y в безопасную зону
        in      a, (PORT_Y) : push af
        ld      a, #C0 : out (PORT_Y), a

        ; 2. Обновить AY регистры (звук)
        ld      a, (SNDFLAG)
        ; ... выбор чипа (TurboSound), вывод AYREGS через ROUT ...

        ; 3. Сохранить/восстановить страницы окон
        in      a, (PAGE1) : ld (_memSlot1), a
        in      a, (PAGE2) : ld (_memSlot2), a
        ld      a, (memSNDPage) : out (PAGE1), a

        ; 4. Вызвать музыкальный плеер (#400C, #4006)
        call    #400C               ; обновить музыку
        call    #4006               ; микшировать

        ; 5. Восстановить страницы
        ld      a, (_memSlot1) : out (PAGE1), a

        ; 6. Счётчик кадров (32-битный)
        ld      hl, _time
        ld      b, 4
.time1: inc     (hl)
        jr      nz, .time2
        inc     hl
        djnz    .time1
.time2:

        ; 7. Восстановить PORT_Y
        pop     af
        out     (PORT_Y), a

        ; 8. Выставить флаг VSync
        ld      a, #FF : ld (vsync_flag), a

        ; 9. Обработать клавиатуру (PS2Scan)
        call    PS2Scan

        ; 10. Восстановить регистры
        pop     hl : pop de : pop bc : pop af
        exx : exa
        pop     iy : pop ix : pop hl : pop de : pop bc : pop af
        ei
        reti
```

### Ключевые детали

- **PORT_Y сохраняется и восстанавливается** — обработчик может менять PORT_Y для работы с палитрой, но обязан вернуть прежнее значение.
- **Страницы окон сохраняются** — обработчик переключает PAGE1/PAGE2 для доступа к музыкальным данным, а потом возвращает.
- **`vsync_flag = #FF`** выставляется в конце — `waitVsync` в главном цикле ждёт именно этот байт.
- **32-битный счётчик `_time`** — инкрементируется каждый кадр (50 Гц → ~20 мс на тик). Можно использовать для отсчёта времени: `_time / 50 = секунды`.

## Ожидание VSync (halt)

```asm
; Главный цикл: рисовать и ждать кадр
GameLoop:
        call    DrawFrame
        halt                        ; ждать следующего VSync
        jr      GameLoop
```

`HALT` работает только при разрешённых прерываниях (`EI`). Z80 останавливается и возобновляется на следующем прерывании.

---

## Приоритеты прерываний

Z80 не имеет аппаратного приоритета — прерывания обрабатываются в порядке поступления, одно за раз. Если во время обработки приходит новое — оно ждёт `EI` + `RETI`.

**Вложенные прерывания** (reentrant) возможны только если в обработчике явно сделать `EI`, но это требует тщательного управления стеком.

## Пример: полный каркас программы с IM2

```asm
        org     #8100

main:
        di

        ; Заполнить таблицу IM2
        ld      hl, #FE00
        ld      de, #FE01
        ld      bc, 256
        ld      (hl), #FD
        ldir

        ; Указатель обработчика на #FDFD
        ld      a, #C3              ; JP opcode
        ld      (#FDFD), a
        ld      hl, int_handler
        ld      (#FDFE), hl

        ld      a, #FE
        ld      i, a
        im      2
        ei

        ; Главный цикл
.loop:
        halt                        ; ждать VSync
        call    GameLogic
        jr      .loop

int_handler:
        push    af
        push    hl

        ld      hl, (tick_count)
        inc     hl
        ld      (tick_count), hl

        pop     hl
        pop     af
        ei
        reti

tick_count:     dw 0

GameLogic:
        ; ... игровая логика ...
        ret
```

## Ключевые моменты

> - **Источники INT:** VSync (~50 Гц), клавиатура PS/2 (~2.1 кГц), CTC (настраиваемый), CBL (COVOX), NMI (Alt+F12).
> - **Формула INT:** `INT_X = !DFF(GND, (SVIDEO.INTT & KEYS.int), ...) or !CBL_INT` — кадровое и клавиатурное через AND → клавиатура вносит джиттер до 460 мкс. Отключается через бит 3 порта `#C3`.
> - **IM 2 трюк с 257 байтами:** таблица из одинаковых `#FD` на `#FE00`–`#FF00`; любой вектор → `#FDFD` → единый обработчик.
> - **waitVsync:** цикл `HALT` + проверка `vsync_flag` — защита от ложного пробуждения (HALT просыпается от любого INT, не только VSync).
> - **Tearing / glitch:** обновляйте VRAM **только в VBI** (после VSync) или используйте double buffering с flip по VSync.
> - **CTC для логики, VSync для графики:** CTC стабилен по частоте, но не синхронизирован с лучом → не пишите в VRAM из CTC-обработчика.
> - Всегда сохраняйте регистры в обработчике, `EI + RETI` в конце.
