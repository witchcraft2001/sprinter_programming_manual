# 5.8 Текстовый режим 80×32

> **Навигация:** [← 5.7 Double buffering](07_double_buffer.md) | [Оглавление](../README.md) | [5.9 Примеры графики →](09_examples.md)

---

## Что это

Текстовый режим Sprinter — это **аппаратный режим видеоконтроллера**. FPGA (`VIDEO2.TDF`) имеет отдельную схему адресации для текстового режима (`VCM[1..0] = 3`), в которой видеоконтроллер самостоятельно:

1. Читает **код символа** и **атрибут** из видеопамяти (VRAM).
2. По коду символа обращается к **знакогенератору** (шрифт), загруженному в VRAM при инициализации BIOS.
3. Формирует пиксели каждой растровой строки символа.

Программа пишет коды символов и атрибуты непосредственно в VRAM — видеоконтроллер растеризует их аппаратно, без участия CPU. Это принципиально быстрее, чем попиксельная отрисовка текста в графическом режиме.

### Роль BIOS

BIOS при инициализации **загружает знакогенератор** (шрифт 256 символов × 8 байт) в выделенную область VRAM. После этого видеоконтроллер работает автономно — CPU только записывает коды и атрибуты.

DSS/BIOS API (`RST #08` / `RST #10`) — удобная обёртка для записи в VRAM, но **прямой доступ к видеопамяти также возможен** и даёт максимальную скорость вывода (см. раздел «Прямой доступ» ниже).

## Характеристики

| Параметр | Значение |
|---------|---------|
| Сетка | **80 столбцов × 32 строки** (2 560 позиций) |
| Шрифт | 8 × 8 пикселей, 256 символов (псевдографика включена) |
| Пиксельное разрешение | 80 × 8 = **640** по горизонтали, 32 × 8 = **256** по вертикали |
| Цвет символа/фона | 4 бита каждый (16 цветов из палитры 16М) |
| Данные в VRAM | Код символа (1 байт) + атрибут (1 байт) на позицию |
| Знакогенератор | 256 × 8 = 2048 байт, загружается BIOS в VRAM |
| Прокрутка | через `Dss.Scroll #55` или прямой LDIR по VRAM |

## Аппаратная схема работы

```
┌──────────┐       ┌───────────────────────────────────┐
│   CPU    │──────►│            VRAM                    │
│ (Z80)    │ пишет │  ┌─────────┐  ┌───────────────┐  │
└──────────┘ коды/ │  │ Символы │  │ Знакогенератор│  │
             атриб.│  │ + Атриб.│  │ (шрифт 2 КБ) │  │
                   │  └────┬────┘  └───────┬───────┘  │
                   └───────┼───────────────┼──────────┘
                           │               │
                    ┌──────▼───────────────▼──────┐
                    │     Видеоконтроллер         │
                    │     (VIDEO2.TDF)             │
                    │                              │
                    │ 1. Читает код символа        │
                    │ 2. Индексирует знакогенератор│
                    │ 3. Читает атрибут (INK/PAPER)│
                    │ 4. Формирует пиксели 8×8     │
                    └──────────┬───────────────────┘
                               │
                          Видеосигнал
```

Из `VIDEO2.TDF` — формирование адреса в текстовом режиме:

```
; VCM case 3 (текстовый режим):
VLA[].d = (DIR_PORT3, CTH[5..0], CT5, VCC, VCC, CTV[8..3], GND, GND);
```

Где:
- `DIR_PORT3` — бит выбора страницы (double buffer)
- `CTH[5..0]` — горизонтальный счётчик (позиция столбца, 0–79)
- `CTV[8..3]` — вертикальный счётчик (позиция строки, 0–31)
- `CT5` — бит строки внутри символа (0–7, для индексации знакогенератора)

Цвет формируется из атрибута аппаратно:

```
; Из VIDEO2.TDF — извлечение INK/PAPER:
WHEN B"1X" => BRVA[7..0] = DCOL[];            ; текстовый режим: полный атрибут
WHEN B"01" => BRVA[7..0] = (B"0000",DCOL[7..4]); ; PAPER (верхний ниббл)
WHEN B"00" => BRVA[7..0] = (B"0000",DCOL[3..0]); ; INK (нижний ниббл)
```

## Структура видеопамяти в текстовом режиме

### Поле графики и положение окна

Видео-ОЗУ Sprinter представляется как **единое поле графики 1024×256 точек** (из документации Peters Plus: `BIOS_FN.TXT`). В знакоместах это **128 столбцов × 32 строки** (по 8×8 пикселей каждое).

На экран выводится только часть этого поля. Текстовое окно 80×32 — это «окно» (viewport) внутри полного поля, расположенное по координатам, заданным при открытии (параметры `IX+6`, `IX+7` дескриптора окна).

```
Поле графики 128×32 знакомест (1024×256 точек)
┌──────────────────────────────────────────────┬──────────────────┐
│                                              │                  │
│       Текстовое окно 80×32                   │   Невидимая      │
│       (то, что видно на экране)              │   область        │
│                                              │   (столбцы       │
│       столбцы 0..79                          │    80..127)      │
│                                              │                  │
│                                              │                  │
└──────────────────────────────────────────────┴──────────────────┘
```

**Куда деваются оставшиеся 48 столбцов (80..127)?** Они физически существуют в VRAM, но не отображаются на экране. Эта область занята:

- **Знакогенераторами.** Каждый знакогенератор занимает 8 столбцов по X (см. раздел «Знакогенератор в поле графики» ниже). До 16 знакогенераторов × 8 столбцов = 128 столбцов — ровно всё поле.
- **Ячейками синхронизации и гашения.** Видеоконтроллер для каждого «квадрата» (знакоместа) хранит в VRAM **байт режима** (`MODE0`). Ячейки за пределами видимой области могут иметь режим **BORDER** (вывод цвета бордюра) или **BLANK** (гашение — чёрный). Эти ячейки участвуют в формировании видеосигнала — бордюра и синхроимпульсов. **Изменять их не следует** — это может привести к нарушению синхронизации или артефактам на экране.

Из `VIDEO2.TDF`:
```
; MODE0[7..4] == %1111 → режим BORDER: вместо символа выводится цвет бордюра
IF DFF((MODE0[7..4] == 15), WR_PIC,,) THEN
    DCOL[].d = (%00, BRD[2..0], BRD[2..0]);    ; цвет бордюра

; BLANK: сигнал гашения обнуляет атрибут
DCOL[].clrn = !BLANK;                           ; чёрный при гашении
```

> **Потенциально** разрешение текстового режима можно увеличить за пределы 80×32 — например, до 96×32 или даже 128×32 — за счёт сужения бордюра. Однако это требует перенастройки BIOS (изменение дескриптора окна и режимов ячеек в VRAM) и не является штатной возможностью. Стандартная конфигурация DSS: окно 80×32 с координатой `(0, 0)` поля графики.

### Адресация: координаты → позиция в памяти

Для программного доступа к текстовому экрану (через BIOS-функции или при работе с буфером `WinCopy`/`WinRest`) используется **линейная адресация**:

```
позиция = столбец + строка × 80
```

Где `столбец` = 0..79, `строка` = 0..31.

Каждая позиция хранит **пару: код символа + атрибут** (по 1 байту каждый). В буфере `WinCopy`/`WinRest` данные идут в формате IBM PC:

```
sym0, atr0, sym1, atr1, sym2, atr2, ...
```

Сначала все пары первой строки (80 пар = 160 байт), затем второй строки и т. д. Общий размер буфера окна 80×32 = **5120 байт** (2560 символов + 2560 атрибутов).

### Аппаратная адресация (VLA)

Видеоконтроллер формирует 18-битный адрес VRAM (`VLA`) из счётчиков развёртки. Для текстового режима (VCM case 3) из `VIDEO2.TDF`:

```
VLA[17..0] = (DIR_PORT3, CTH[5..0], CT5, VCC, VCC, CTV[8..3], GND, GND)
```

Разбор по битам:

```
Бит VLA  Источник         Назначение
──────────────────────────────────────────────────
  17     DIR_PORT3         Выбор страницы (double buffer)
  16..11 CTH[5..0]         Номер столбца (0..79)
  10     CT5               Строка растра внутри символа (для знакогенератора)
   9..8  VCC, VCC          Константы = %11
   7..2  CTV[8..3]         Номер строки (0..31)
   1..0  GND, GND          Константы = %00
```

Из этого следует аппаратный адрес символа в VRAM:

```
VRAM_адрес = (страница << 17) | (столбец << 11) | (%11 << 8) | (строка << 2)
```

Или проще: видеоконтроллер обращается к VRAM по «разбросанным» адресам — столбец в старших битах, строка в младших. Это **не** линейный буфер типа `строка × 80 + столбец`, а аппаратная раскладка, оптимизированная под счётчики развёртки.

### Знакогенератор в поле графики

Знакогенератор (шрифт) **физически хранится в том же поле графики** VRAM. Его положение определяется номером в описателе окна (биты `IX+4`):

```
X_зг = (8 × номер_зг[3..0]) .. (8 × номер_зг[3..0] + 7)   ; 8 знакомест по X
Y_зг = 0 .. 31                                                ; все строки по Y
```

Если знакогенератор №0, он занимает столбцы 0–7 поля графики. Текстовое окно 80×32 занимает столбцы 0–79 — они **пересекаются**. Именно поэтому знакогенератор «встроен» в экранные данные: его битовые образы символов лежат в тех же VRAM-ячейках, которые видеоконтроллер интерпретирует и как данные шрифта.

> **Практический вывод:** при прямой записи в VRAM не записывайте в область, занятую знакогенератором — иначе «испортите» шрифт. Стандартный знакогенератор №0 занимает первые 8 столбцов, поэтому символы с координатами столбцов 0–7 содержат данные шрифта.

### Пример: расчёт адреса для WinCopy-буфера

```asm
; Вычислить смещение символа (col, row) в буфере WinCopy/WinRest
; Вход: D = row (0..31), E = col (0..79)
; Выход: HL = смещение в буфере (sym байт); HL+1 = атрибут

CalcBufOffset:
        ld      h, 0
        ld      l, d                ; L = row
        ; HL = row × 80: row × 64 + row × 16
        add     hl, hl              ; × 2
        add     hl, hl              ; × 4
        add     hl, hl              ; × 8
        add     hl, hl              ; × 16
        ld      b, h
        ld      c, l                ; BC = row × 16
        add     hl, hl              ; × 32
        add     hl, hl              ; × 64
        add     hl, bc              ; HL = row × 80
        ld      b, 0
        ld      c, e                ; BC = col
        add     hl, bc              ; HL = row × 80 + col
        add     hl, hl              ; × 2 (каждая позиция = 2 байта: sym + atr)
        ret
        ; HL = смещение в буфере
        ; (buf + HL) = код символа
        ; (buf + HL + 1) = атрибут
```

### Пример: прямая запись символа через BIOS

```asm
; Вместо расчёта адреса VRAM — использовать BIOS Win_Put_Sym (#B5)
; Он сам вычислит правильный аппаратный адрес

        ld      a, 0                ; ID окна (всегда 0)
        ld      d, 5                ; row
        ld      e, 10               ; col
        ld      l, 'A'              ; символ
        ld      h, #0F              ; атрибут (белый на чёрном)
        ld      b, 0                ; номер знакогенератора
        ld      c, #B5              ; Win_Put_Sym
        rst     #08
```

## Включение текстового режима

При старте из DSS текстовый режим обычно уже активен. Если нужно переключиться явно:

```asm
Dss.SetVMod     EQU #50
Dss.GetVMod     EQU #51

; Узнать текущий режим
        ld      c, Dss.GetVMod
        rst     #10
        ; A = код режима, B = видеостраница

; Сохранить и восстановить при необходимости
        ld      (saved_mode), a
        ; ...
        ld      a, (saved_mode)
        ld      c, Dss.SetVMod
        rst     #10
```

## Два способа вывода текста

### Способ 1: через DSS/BIOS API (удобно)

Все функции — через `RST #10`, номер в регистре `C`:

| Код | Имя | Назначение |
|-----|-----|-----------|
| `#52` | `Locate` | Установить позицию курсора (D = row, E = col) |
| `#53` | `Cursor` | Получить позицию курсора (→ D, E) |
| `#55` | `Scroll` | Прокрутить окно (B = направление, D/E = поз., H/L = размер) |
| `#56` | `Clear` | Очистить окно (A = символ, B = атрибут, D/E = поз., H/L = размер) |
| `#57` | `RdChar` | Прочитать символ в позиции (→ A = символ, B = атрибут) |
| `#58` | `WrChar` | Записать символ с атрибутом (A = символ, B = атрибут, D/E = поз.) |
| `#59` | `WinCopy` | Скопировать окно в буфер |
| `#5A` | `WinRest` | Восстановить окно из буфера |
| `#5B` | `PutChar` | Печать символа (с обработкой CR, LF, Tab и автоскроллом) |
| `#5C` | `PChars` | Печать ASCIIZ-строки (HL = адрес строки) |

### Способ 2: прямой доступ к VRAM (быстро)

Программа может записывать коды символов и атрибуты **непосредственно в VRAM** — видеоконтроллер отобразит их аппаратно. Это значительно быстрее, чем вызовы BIOS, и применяется в программах с большим объёмом текстового вывода.

Пример из `LION.ASZ` — текстовый редактор, работающий через прямую запись в VRAM:

```asm
; Пример: прямой вывод строки в видеопамять текстового режима
; HL = адрес строки, DE = позиция в VRAM, B = длина, E' = атрибут

DirectPrintLine:
        in      a, (#A2)            ; сохранить WIN1
        push    af
        ld      a, #50              ; VPAGE_TILES
        out     (#A2), a            ; WIN1 = VRAM

        ; Для каждого символа:
        ;   VRAM[позиция] = код символа
        ;   VRAM[позиция + смещение] = атрибут
        ; (точная раскладка зависит от того, как видеоконтроллер
        ;  адресует символы и атрибуты в VLA)

.loop:
        ld      a, (hl)             ; код символа
        ld      (de), a             ; записать в VRAM
        inc     hl
        inc     de
        djnz    .loop

        pop     af
        out     (#A2), a            ; восстановить WIN1
        ret
```

Через BIOS (`RST #08`, функция `CMD_PRINT_LINE3 = #87`) тот же вывод:

```asm
; Вывод через BIOS (из LION.ASZ)
        ld      hl, text_data       ; адрес строки
        ld      b, line_length      ; длина
        ld      e, #29              ; атрибут (цвет)
        ld      d, #0D              ; delimiter (CR)
        ld      c, #87              ; BIOS LP_PRINT_LN3
        rst     #08
```

## Атрибут символа

Байт атрибута кодирует цвет текста и фона:

| Биты | Поле | Диапазон |
|------|------|---------|
| [3:0] | INK (цвет символа) | 0–15 |
| [7:4] | PAPER (цвет фона) | 0–15 |

16 цветов выбираются из палитры 16 777 216 (6-6-6 RGB). Первые 16 записей при старте установлены в ZX-подобные цвета (см. [5.5 Палитра](05_palette.md)).

Пример атрибутов:
- `#0F` — белый текст на чёрном фоне
- `#17` — белый текст на синем
- `#29` — циановый текст на чёрном (из LION.ASZ)
- `#F0` — чёрный текст на белом
- `#4E` — жёлтый текст на красном

## Базовые примеры

### Hello, World через DSS API

```asm
        org     #8100 - 512
        db      "EXE", 0
        dw      #0200, #0200, 0, #8100, main, #BFFF
        ds      512 - 16

        org     #8100
main:
        ; Установить курсор в (row=5, col=10)
        ld      d, 5
        ld      e, 10
        ld      c, #52              ; Locate
        rst     #10

        ; Напечатать ASCIIZ строку
        ld      hl, msg
        ld      c, #5C              ; PChars
        rst     #10

        ld      c, #30              ; WaitKey
        rst     #10
        ld      c, #41              ; Exit
        rst     #10

msg:    db      "Hello from text mode!", 13, 10, 0
```

### Цветной символ

```asm
; Записать символ '*' в позицию (row=10, col=20) с атрибутом yellow-on-red
        ld      a, '*'
        ld      b, #4E              ; PAPER=4 (red), INK=14 (yellow)
        ld      d, 10
        ld      e, 20
        ld      c, #58              ; WrChar
        rst     #10
```

### Очистка окна с цветом

```asm
; Очистить прямоугольник 30×5 от (row=2, col=10) синим фоном
        ld      a, ' '
        ld      b, #17              ; PAPER=1 (blue), INK=7 (gray)
        ld      d, 2
        ld      e, 10
        ld      h, 5
        ld      l, 30
        ld      c, #56              ; Clear
        rst     #10
```

### Сохранение и восстановление области экрана

```asm
; Для «всплывающих» диалогов
SaveArea:
        ld      d, 5 : ld e, 20
        ld      h, 10 : ld l, 40
        ld      ix, save_buffer
        ld      b, 0
        ld      c, #59              ; WinCopy
        rst     #10
        ret

RestoreArea:
        ld      d, 5 : ld e, 20
        ld      h, 10 : ld l, 40
        ld      ix, save_buffer
        ld      b, 0
        ld      c, #5A              ; WinRest
        rst     #10
        ret

save_buffer:    ds 40 * 10 * 2      ; символ + атрибут × площадь
```

## Курсор

```asm
; Установить позицию
        ld      d, <row>            ; 0..31
        ld      e, <col>            ; 0..79
        ld      c, #52
        rst     #10

; Получить позицию
        ld      c, #53
        rst     #10
        ; D = row, E = col
```

> DSS не рисует мигающий курсор — его отображение возлагается на приложение (обычно через `WrChar` с инвертированным атрибутом и восстановление по таймеру).

## Спецсимволы и управление

Функции `PutChar` (`#5B`) и `PChars` (`#5C`) обрабатывают управляющие коды:

| Код | Действие |
|-----|---------|
| 13 (CR) | Возврат каретки (курсор в колонку 0) |
| 10 (LF) | Перевод строки (row++) |
| 9 (Tab) | Табуляция до следующей границы 8 |
| ≥ 32 | Обычный символ, печать с инкрементом колонки |

При переходе за правый край — автоперенос; при достижении низа — автопрокрутка.

## Псевдографика

Знакогенератор включает 256 символов с псевдографикой IBM CP866/CP437:

| Код | Символ | Назначение |
|-----|--------|-----------|
| `#C9` | ╔ | Левый верхний угол |
| `#BB` | ╗ | Правый верхний угол |
| `#C8` | ╚ | Левый нижний |
| `#BC` | ╝ | Правый нижний |
| `#BA` | ║ | Вертикаль |
| `#CD` | ═ | Горизонталь |

```asm
; Нарисовать рамку 40×10 от (row=5, col=10)
DrawFrame:
        ld      d, 5 : ld e, 10
        ld      c, #52 : rst #10     ; Locate

        ld      a, #C9               ; ╔
        ld      c, #5B : rst #10

        ld      b, 38
.top:
        ld      a, #CD               ; ═
        push    bc
        ld      c, #5B : rst #10
        pop     bc
        djnz    .top

        ld      a, #BB               ; ╗
        ld      c, #5B : rst #10
        ; Боковые стороны и нижняя — аналогично
        ret
```

## Пользовательский знакогенератор (шрифт)

BIOS позволяет **заменить стандартный шрифт** на свой через функцию `Win_Set_ZG` (`#B6`, `RST #08`). Это загружает новый знакогенератор в VRAM, и видеоконтроллер сразу начинает использовать его для отображения.

### BIOS API знакогенератора

| Код | Имя | Назначение |
|-----|-----|-----------|
| `#B6` | `Win_Set_ZG` | Загрузить пользовательский знакогенератор в VRAM |
| `#B8` | `Win_Get_ZG` | Получить текущий знакогенератор из VRAM |

### Формат знакогенератора

Знакогенератор — массив **256 символов × 8 байт** = **2048 байт**. Каждый символ — 8 строк по 8 пикселей (1 бит = 1 пиксель, старший бит — левый):

```
; Символ 'A' (код #41) — 8 байт:
;   Строка 0: %00011000  = #18
;   Строка 1: %00100100  = #24
;   Строка 2: %01000010  = #42
;   Строка 3: %01111110  = #7E
;   Строка 4: %01000010  = #42
;   Строка 5: %01000010  = #42
;   Строка 6: %01000010  = #42
;   Строка 7: %00000000  = #00
```

### Пример: загрузить свой шрифт

```asm
Bios.Win_Set_ZG EQU #B6

; Загрузить пользовательский шрифт
; HL = адрес массива 2048 байт (256 символов × 8)
LoadCustomFont:
        ld      hl, my_font
        ld      c, Bios.Win_Set_ZG
        rst     #08
        ret

my_font:
        ; 256 символов × 8 байт = 2048 байт
        ; Символ 0:
        db  #00, #00, #00, #00, #00, #00, #00, #00
        ; Символ 1:
        db  #7E, #81, #A5, #81, #BD, #99, #81, #7E
        ; ... ещё 254 символа ...
```

### Пример: изменить один символ

Если нужно изменить только один символ, можно:
1. Получить текущий шрифт через `Win_Get_ZG` (`#B8`)
2. Модифицировать нужные 8 байт в буфере
3. Загрузить обратно через `Win_Set_ZG` (`#B6`)

```asm
Bios.Win_Get_ZG EQU #B8

; Заменить символ #80 на «стрелку вправо»
ModifyOneChar:
        ; 1. Получить текущий шрифт
        ld      hl, font_buf
        ld      c, Bios.Win_Get_ZG
        rst     #08

        ; 2. Вычислить смещение символа #80: 128 × 8 = 1024
        ld      hl, font_buf + 1024

        ; 3. Записать новый глиф
        ld      (hl), %00000000 : inc hl
        ld      (hl), %00001000 : inc hl
        ld      (hl), %00001100 : inc hl
        ld      (hl), %01111110 : inc hl
        ld      (hl), %01111110 : inc hl
        ld      (hl), %00001100 : inc hl
        ld      (hl), %00001000 : inc hl
        ld      (hl), %00000000

        ; 4. Загрузить обратно
        ld      hl, font_buf
        ld      c, Bios.Win_Set_ZG
        rst     #08
        ret

font_buf:   ds 2048
```

### Применение

- Кириллица / латиница с другой раскладкой
- Специальные символы для игр (мини-тайлы 8×8)
- Математические или технические обозначения
- Шрифт другого размера (визуально — через модификацию глифов)

## Когда использовать текстовый режим

**Используйте**, если:
- Приложение — утилита, редактор, настройки, отладчик.
- Нужен быстрый вывод текста — видеоконтроллер рендерит аппаратно.
- Достаточно 80×32 сетки с 16 цветами.
- Нужна прокрутка, окна, сохранение/восстановление — через API «из коробки».
- Возможен прямой доступ к VRAM для максимальной скорости.

**Не используйте**, если:
- Нужна пиксельная графика (спрайты, линии) — тогда 320×256 или 640×256.
- Нужно более 16 одновременных цветов.

## Ключевые моменты

> - Текстовый режим — **аппаратный**: видеоконтроллер сам читает коды/атрибуты из VRAM и рендерит через знакогенератор.
> - Сетка 80×32 (80 × 8 = 640 пикселей, 32 × 8 = 256 пикселей), 16-цветный атрибут INK|PAPER.
> - BIOS при старте загружает знакогенератор (шрифт 2 КБ) в VRAM — далее CPU не участвует в растеризации.
> - Два способа вывода: через DSS API (`#52`–`#5C`) или **прямой записью в VRAM** (быстрее).
> - По умолчанию при старте из DSS программа уже в текстовом режиме.
> - Псевдографика CP866 для рамок и диалогов.
> - `WinCopy`/`WinRest` для всплывающих окон.
